# 第9章 比例份额调度：彩票调度实现总结

## 一、本节主题

本节主要讲 **彩票调度的具体实现方式**。

彩票调度的核心思想是：

> 每个进程拥有一定数量的彩票，彩票数越多，被调度运行的概率越大。

在实现上，彩票调度并不复杂，只需要：

1. 一个随机数生成器；
2. 一个记录进程信息的数据结构；
3. 每个进程的彩票数；
4. 系统中的总彩票数。

------

## 二、基本数据结构

假设系统中有 3 个进程：

| 进程 | 彩票数 |
| ---- | ------ |
| A    | 100    |
| B    | 50     |
| C    | 250    |

总彩票数为：

```c
totaltickets = 400;
```

那么这 3 个进程对应的彩票区间可以理解为：

| 进程 | 彩票区间  |
| ---- | --------- |
| A    | 0 ~ 99    |
| B    | 100 ~ 149 |
| C    | 150 ~ 399 |

调度器每次会从 `0 ~ 399` 中随机抽取一个中奖号码。

中奖号码落在哪个进程的彩票区间中，哪个进程就获得 CPU。

------

## 三、核心代码

彩票调度的核心实现代码如下：

```c
// counter: used to track if we've found the winner yet
int counter = 0;

// winner: use some call to a random number generator to
// get a value, between 0 and the total # of tickets
int winner = getrandom(0, totaltickets);

// current: use this to walk through the list of jobs
node_t *current = head;

// loop until the sum of ticket values is > the winner
while (current) {
    counter = counter + current->tickets;
    if (counter > winner)
        break; // found the winner
    current = current->next;
}

// 'current' is the winner: schedule it...
```

------

## 四、代码执行过程

假设中奖号码为：

```c
winner = 300;
```

进程彩票情况如下：

| 进程 | 彩票数 |
| ---- | ------ |
| A    | 100    |
| B    | 50     |
| C    | 250    |

执行过程如下：

### 1. 遍历进程 A

```c
counter = 0 + 100 = 100;
```

判断：

```c
100 > 300
```

不成立，所以 A 没有中奖，继续遍历。

------

### 2. 遍历进程 B

```c
counter = 100 + 50 = 150;
```

判断：

```c
150 > 300
```

不成立，所以 B 没有中奖，继续遍历。

------

### 3. 遍历进程 C

```c
counter = 150 + 250 = 400;
```

判断：

```c
400 > 300
```

成立，所以 C 中奖。

最终调度器选择进程 C 运行。

------

## 五、为什么用 `counter > winner`

`counter` 表示已经遍历过的进程的彩票累计数量。

当 `counter` 第一次大于 `winner` 时，说明中奖号码落在当前进程的彩票区间中。

例如：

| 进程 | 彩票区间  | counter 变化 |
| ---- | --------- | ------------ |
| A    | 0 ~ 99    | 100          |
| B    | 100 ~ 149 | 150          |
| C    | 150 ~ 399 | 400          |

如果 `winner = 300`，那么它落在 C 的区间 `150 ~ 399` 中。

因此，当遍历到 C 时：

```c
counter = 400;
```

此时：

```c
counter > winner
```

成立，所以 C 是中奖者。

------

## 六、算法复杂度

如果系统中有 N 个进程，并且使用链表存储进程，那么最坏情况下需要遍历整个链表。

因此时间复杂度为：

```text
O(N)
```

最坏情况是中奖进程刚好在链表最后一个位置。

------

## 七、排序优化

为了提高查找中奖进程的效率，可以将进程列表按照彩票数从大到小排序。

例如原来的顺序是：

```text
A: 10 张
B: 20 张
C: 1000 张
```

可以调整为：

```text
C: 1000 张
B: 20 张
A: 10 张
```

这样做的好处是：

> 彩票数多的进程中奖概率更大，把它们放在前面，可以减少平均遍历次数。

需要注意的是：

```text
排序不会影响算法的正确性，只会影响查找效率。
```

即使不排序，彩票调度仍然是正确的。

------

## 八、本节重点

本节需要掌握以下内容：

1. 彩票调度的实现非常简单；
2. 每个进程拥有一定数量的彩票；
3. 调度器随机生成一个中奖号码；
4. 通过遍历进程列表并累加彩票数来找到中奖进程；
5. `counter > winner` 表示中奖号码落在当前进程的彩票区间中；
6. 使用链表实现时，时间复杂度为 `O(N)`；
7. 按彩票数递减排序可以提高平均查找效率。

------

## 九、考试常见考点

### 1. 给出彩票数和中奖号码，判断哪个进程运行

例如：

| 进程 | 彩票数 |
| ---- | ------ |
| A    | 100    |
| B    | 50     |
| C    | 250    |

如果：

```c
winner = 300;
```

则计算：

```text
A: counter = 100
B: counter = 150
C: counter = 400
```

因为：

```text
400 > 300
```

所以 C 运行。

------

### 2. 解释 `counter > winner` 的含义

答案：

> `counter` 是当前已经累计的彩票数，当它第一次超过中奖号码时，说明中奖号码落在当前进程的彩票范围内，因此当前进程就是中奖者。

------

### 3. 分析彩票调度的时间复杂度

答案：

> 如果使用链表存储进程，查找中奖者最坏情况下需要遍历所有进程，因此时间复杂度为 `O(N)`。

------

### 4. 为什么要按照彩票数递减排序

答案：

> 按彩票数递减排序可以让彩票数多、中奖概率高的进程排在前面，从而减少平均遍历次数，提高调度决策效率。但排序不会影响算法正确性。

------

## 十、一句话总结

彩票调度的实现就是：

> 把所有进程的彩票看成一段连续区间，随机抽取一个中奖号码，然后遍历进程列表，找到这个号码落在哪个进程的区间中，该进程就获得 CPU。
