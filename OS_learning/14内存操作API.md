# 第14章 插叙：内存操作 API

## 一、本章核心问题

> 在 UNIX/C 程序中，如何正确地分配和管理内存？

本章主要介绍 C 语言中和内存分配相关的 API，包括：

- 栈内存和堆内存的区别
- `malloc()` 如何申请堆内存
- `free()` 如何释放堆内存
- 常见的内存错误
- `malloc/free` 背后的操作系统支持
- `calloc()` 和 `realloc()` 等其他接口

这一章虽然短，但非常重要。因为 C/C++ 程序里大量 bug 都和内存管理有关。

------

# 14.1 内存类型

C 程序运行时，常见的内存可以分为两类：

## 1. 栈内存

栈内存由编译器自动管理，也叫自动内存。

例如：

```c
void func() {
    int x;
}
```

这里的 `x` 是一个局部变量，存放在栈上。

当函数被调用时，编译器自动为它分配空间；当函数返回时，这块空间自动释放。

### 栈内存特点

- 由编译器自动申请和释放
- 生命周期通常只在函数调用期间
- 使用方便
- 不适合保存函数返回后还要继续使用的数据

错误示例：

```c
int* func() {
    int x = 10;
    return &x;
}
```

`x` 是栈变量，函数返回后空间已经失效，返回它的地址是危险的。

------

## 2. 堆内存

堆内存需要程序员手动申请和释放。

例如：

```c
void func() {
    int *x = malloc(sizeof(int));
}
```

这里有两部分：

```c
int *x;
```

指针变量 `x` 本身通常在栈上。

```c
malloc(sizeof(int));
```

这会在堆上申请一块可以存放 `int` 的空间，并返回这块空间的地址。

所以：

- 指针变量 `x` 在栈上
- `x` 指向的内存在堆上

### 堆内存特点

- 由程序员显式申请
- 由程序员显式释放
- 生命周期更灵活
- 容易出现内存泄漏、重复释放、悬挂指针等问题

------

# 14.2 malloc() 调用

## 1. malloc 的作用

`malloc()` 用于在堆上申请一块指定大小的内存。

函数原型：

```c
#include <stdlib.h>

void *malloc(size_t size);
```

含义：

- `size`：要申请的字节数
- 成功：返回指向新内存区域的指针
- 失败：返回 `NULL`

例如：

```c
double *d = malloc(sizeof(double));
```

这表示在堆上申请一块可以存放 `double` 的空间。

------

## 2. 为什么常用 sizeof

不推荐直接写数字：

```c
malloc(8);
```

更推荐：

```c
malloc(sizeof(double));
```

原因是不同机器、不同数据类型大小可能不同，使用 `sizeof` 更安全。

申请 10 个整数：

```c
int *arr = malloc(10 * sizeof(int));
```

这表示申请可以存放 10 个 `int` 的堆空间。

------

## 3. sizeof 的常见陷阱

看下面代码：

```c
int *x = malloc(10 * sizeof(int));
printf("%lu\n", sizeof(x));
```

很多人以为 `sizeof(x)` 会输出 40，但实际上不会。

因为 `x` 是一个指针，`sizeof(x)` 得到的是指针本身的大小。

通常：

- 32 位机器上指针大小是 4 字节
- 64 位机器上指针大小是 8 字节

它不是堆数组的总大小。

------

## 4. 普通数组中的 sizeof

如果是普通数组：

```c
int x[10];
printf("%lu\n", sizeof(x));
```

这时 `sizeof(x)` 通常会得到 40。

原因是编译器知道 `x` 是一个包含 10 个 `int` 的数组。

所以要区分：

```c
sizeof(数组名)
```

可能得到整个数组大小。

```c
sizeof(指针)
```

只能得到指针本身大小。

------

## 5. 字符串分配要多加 1

给字符串申请空间时，要给字符串结束符 `\0` 留位置。

例如：

```c
char *p = malloc(strlen(s) + 1);
```

这里的 `+1` 非常重要。

字符串 `"hello"` 看起来长度是 5，但实际存储需要 6 个字符：

```text
'h' 'e' 'l' 'l' 'o' '\0'
```

如果忘记 `+1`，就可能导致越界写入。

------

## 6. malloc 返回 void *

`malloc()` 返回的是：

```c
void *
```

`void *` 是通用指针，可以转换成任意类型的指针。

在 C 语言中，下面两种写法都能见到：

```c
int *x = (int *)malloc(sizeof(int));
```

或者：

```c
int *x = malloc(sizeof(int));
```

在 C 中，强制转换 `malloc()` 的返回值不是必须的。

------

# 14.3 free() 调用

## 1. free 的作用

`free()` 用于释放之前通过 `malloc()` 申请的堆内存。

例如：

```c
int *x = malloc(10 * sizeof(int));

free(x);
```

`free()` 接收一个参数，这个参数必须是由 `malloc()`、`calloc()` 或 `realloc()` 返回的合法堆指针。

------

## 2. free 为什么不用传大小

申请内存时要写大小：

```c
malloc(10 * sizeof(int));
```

释放时却只需要：

```c
free(x);
```

原因是内存分配器会在内部记录这块内存的大小。

通常，分配器会在用户可见内存附近保存一些元数据，例如：

- 这块内存有多大
- 是否已经被分配
- 相邻块信息

所以 `free(x)` 时，分配器可以通过 `x` 找到对应的管理信息。

------

# 14.4 常见错误

C 语言内存错误的特点是：

> 能编译通过，不代表程序正确；运行过一次，也不代表程序正确。

很多内存错误具有偶然性，有时正常运行，有时崩溃，有时悄悄破坏数据。

------

## 1. 忘记分配内存

错误示例：

```c
char *src = "hello";
char *dst;        // 没有分配内存
strcpy(dst, src); // 可能段错误
```

问题在于：

```c
char *dst;
```

只是声明了一个指针变量，但它没有指向一块合法内存。

正确写法：

```c
char *src = "hello";
char *dst = malloc(strlen(src) + 1);

strcpy(dst, src);
```

也可以使用：

```c
char *dst = strdup(src);
```

`strdup()` 会自动分配内存并复制字符串。

------

## 2. 没有分配足够的内存

错误示例：

```c
char *src = "hello";
char *dst = malloc(strlen(src)); // 太小了
strcpy(dst, src);
```

`strlen("hello")` 是 5，但字符串还需要一个 `\0`，所以实际需要 6 个字节。

正确写法：

```c
char *dst = malloc(strlen(src) + 1);
```

这类错误叫缓冲区溢出。

缓冲区溢出可能导致：

- 程序崩溃
- 覆盖其他变量
- 修改堆管理信息
- 安全漏洞
- 随机错误

------

## 3. 忘记初始化分配的内存

`malloc()` 只负责分配空间，不会自动清零。

错误示例：

```c
int *p = malloc(sizeof(int));

printf("%d\n", *p);
```

这里 `*p` 的值是不确定的。

正确写法：

```c
int *p = malloc(sizeof(int));
*p = 10;
```

或者使用 `calloc()`：

```c
int *p = calloc(1, sizeof(int));
```

`calloc()` 会分配内存，并把内容初始化为 0。

------

## 4. 忘记释放内存

如果使用 `malloc()` 申请了内存，但用完后没有 `free()`，就会造成内存泄漏。

错误示例：

```c
void func() {
    int *p = malloc(sizeof(int));
    *p = 10;
}
```

函数结束后：

- 指针变量 `p` 消失了
- 但堆上的内存还在
- 之后再也无法访问这块内存
- 这就是内存泄漏

正确写法：

```c
void func() {
    int *p = malloc(sizeof(int));
    *p = 10;

    free(p);
}
```

------

## 5. 在用完之前释放内存

如果过早释放内存，但之后又继续使用这个指针，就会形成悬挂指针。

错误示例：

```c
int *p = malloc(sizeof(int));
*p = 10;

free(p);

printf("%d\n", *p); // 错误
```

`free(p)` 后，`p` 指向的内存已经不再属于程序使用，但 `p` 本身还保存着原来的地址。

更安全的写法：

```c
free(p);
p = NULL;
```

------

## 6. 反复释放内存

重复释放同一块内存叫 double free。

错误示例：

```c
int *p = malloc(sizeof(int));

free(p);
free(p); // 错误
```

重复释放的结果是未定义的，可能导致：

- 程序崩溃
- 堆结构损坏
- 安全漏洞
- 随机错误

释放后可以把指针设为 `NULL`：

```c
free(p);
p = NULL;
```

因为 `free(NULL)` 是安全的。

------

## 7. 错误地调用 free()

`free()` 只能释放由 `malloc()`、`calloc()`、`realloc()` 返回的堆内存指针。

错误示例 1：释放栈变量地址

```c
int x = 10;
free(&x); // 错误
```

错误示例 2：释放字符串常量

```c
char *s = "hello";
free(s); // 错误
```

错误示例 3：释放偏移后的指针

```c
int *p = malloc(10 * sizeof(int));

free(p + 1); // 错误
```

正确写法：

```c
free(p);
```

------

## 8. 为什么进程退出时没有真正的内存泄漏

对于短时间运行的程序，如果没有调用 `free()`，程序退出后操作系统会回收整个进程的地址空间。

系统中有两级内存管理：

### 第一级：操作系统管理进程地址空间

操作系统负责给进程分配内存页。

进程结束时，操作系统会回收：

- 代码段
- 数据段
- 栈
- 堆
- 映射区域

### 第二级：进程内部的堆管理器

`malloc()` 和 `free()` 管理的是进程内部的堆空间。

如果没调用 `free()`，只是进程内部的堆管理器没有回收这块空间。

但整个进程结束时，操作系统会直接回收所有页面。

不过，对于长期运行的程序，例如 Web 服务器、数据库、操作系统内核，内存泄漏会越来越严重，最终可能导致程序崩溃。

## 9. 内存错误检测工具

常见工具包括：

- `valgrind`
- `purify`
- `AddressSanitizer`

其中 `valgrind` 很常用。

使用方式：

```bash
valgrind ./a.out
```

检查内存泄漏：

```bash
valgrind --leak-check=full ./a.out
```

# 14.5 底层操作系统支持

## 1. malloc/free 不是系统调用

`malloc()` 和 `free()` 本身不是系统调用，而是 C 库函数。

它们主要负责管理进程虚拟地址空间中的堆区域。

也就是说：

```text
用户程序
   |
   | 调用 malloc/free
   v
C 库内存分配器
   |
   | 必要时调用 brk / sbrk / mmap
   v
操作系统
```

小块内存分配可能只在用户态完成。

当堆空间不够时，内存分配器才会通过系统调用向操作系统申请更多内存。

------

## 2. brk 和 sbrk

`brk()` 用来改变程序 break 的位置。

program break 可以理解为：

```text
堆的结束位置
```

如果 break 往高地址移动，堆变大。

如果 break 往低地址移动，堆变小。

示意图：

```text
低地址
+----------------+
| 代码段          |
+----------------+
| 数据段          |
+----------------+
| 堆              |
|                |
|                |  <- program break
+----------------+
| 空闲区域        |
+----------------+
| 栈              |
+----------------+
高地址
```

`brk()` 直接指定新的 break 地址：

```c
brk(new_address);
```

`sbrk()` 传入一个增量：

```c
sbrk(increment);
```

例如：

```c
sbrk(4096);
```

表示请求堆增加 4096 字节。

普通程序员不应该直接使用 `brk()` 或 `sbrk()`，因为它们可能破坏 `malloc` 库内部的堆管理结构。

------

## 3. mmap 获取内存

`mmap()` 也可以向操作系统申请内存。

它可以用于：

- 文件映射
- 匿名内存映射

匿名映射表示这段内存不对应具体文件，可以像堆内存一样使用。

某些 `malloc` 实现会在申请大块内存时使用 `mmap()`，而不是通过 `brk/sbrk` 扩展堆。

可以粗略理解为：

```text
小块内存：malloc 可能从堆里切
大块内存：malloc 可能直接 mmap 一块新区域
```

------

# 14.6 其他调用

除了 `malloc()` 和 `free()`，内存分配库还提供了其他接口。

------

## 1. calloc()

`calloc()` 的作用是：

```text
分配内存，并把内存初始化为 0
```

函数形式：

```c
void *calloc(size_t num, size_t size);
```

例如：

```c
int *arr = calloc(10, sizeof(int));
```

表示申请 10 个 `int` 的空间，并把所有字节初始化为 0。

可以粗略理解为：

```c
int *arr = malloc(10 * sizeof(int));
memset(arr, 0, 10 * sizeof(int));
```

但 `calloc()` 更方便，也更不容易忘记初始化。

### malloc 和 calloc 对比

| 对比项         | malloc     | calloc                   |
| -------------- | ---------- | ------------------------ |
| 是否分配内存   | 是         | 是                       |
| 是否初始化为 0 | 否         | 是                       |
| 参数形式       | 总字节数   | 元素个数 + 每个元素大小  |
| 常见用途       | 普通堆分配 | 数组分配、需要清零的内存 |

------

## 2. realloc()

`realloc()` 用于调整之前分配的内存大小。

函数形式：

```c
void *realloc(void *ptr, size_t new_size);
```

例如：

```c
int *arr = malloc(10 * sizeof(int));

arr = realloc(arr, 20 * sizeof(int));
```

表示把原来能放 10 个 `int` 的数组扩展成能放 20 个 `int`。

------

## 3. realloc 可能做什么

### 情况一：原地扩展

如果原来的内存块后面刚好有足够空间，`realloc()` 可以直接扩展。

```text
原内存块后面空间足够
=> 原地变大
=> 返回地址可能不变
```

### 情况二：重新分配

如果原来的位置放不下更大的空间，`realloc()` 可能会：

1. 重新申请一块更大的内存
2. 把旧内容复制过去
3. 释放旧内存
4. 返回新地址

```text
原位置空间不够
=> 找新位置
=> 复制旧数据
=> 释放旧块
=> 返回新指针
```

所以 `realloc()` 返回的新地址可能和原地址不同。

------

## 4. realloc 的安全写法

不推荐直接写：

```c
arr = realloc(arr, new_size);
```

因为如果 `realloc()` 失败，会返回 `NULL`，原来的 `arr` 地址就丢了，容易造成内存泄漏。

更安全的写法：

```c
int *tmp = realloc(arr, new_size);

if (tmp == NULL) {
    // 扩容失败，arr 仍然有效
    // 可以进行错误处理
} else {
    arr = tmp;
}
```

------

# 14.7 小结

本章介绍了 C 语言中常见的内存分配 API。

| 接口        | 作用                     |
| ----------- | ------------------------ |
| `malloc()`  | 申请一块未初始化的堆内存 |
| `free()`    | 释放之前申请的堆内存     |
| `calloc()`  | 申请内存并初始化为 0     |
| `realloc()` | 调整已有内存块大小       |
| `brk/sbrk`  | 底层改变堆大小的系统调用 |
| `mmap()`    | 创建新的内存映射区域     |

# 本章重点总结

## 1. 栈和堆的区别

| 对比项   | 栈内存           | 堆内存                   |
| -------- | ---------------- | ------------------------ |
| 管理者   | 编译器           | 程序员                   |
| 申请方式 | 声明局部变量     | `malloc()`               |
| 释放方式 | 函数返回自动释放 | `free()`                 |
| 生命周期 | 通常较短         | 更灵活                   |
| 常见风险 | 返回局部变量地址 | 泄漏、重复释放、悬挂指针 |

------

## 2. malloc/free 的本质

`malloc()` 和 `free()` 是库函数，不是系统调用。

它们在进程内部管理堆空间。

当堆空间不够时，内存分配器会通过：

```text
brk / sbrk / mmap
```

向操作系统申请更多内存。

------

## 3. 常见内存错误总表

| 错误类型     | 说明                       | 典型后果   |
| ------------ | -------------------------- | ---------- |
| 忘记分配内存 | 指针没有指向合法空间就使用 | 段错误     |
| 分配空间不足 | 没有给数据留够空间         | 缓冲区溢出 |
| 忘记初始化   | 读取未知值                 | 随机错误   |
| 忘记释放     | `malloc` 后没有 `free`     | 内存泄漏   |
| 过早释放     | `free` 后继续使用          | 悬挂指针   |
| 重复释放     | 对同一块内存多次 `free`    | 堆损坏     |
| 错误释放     | `free` 非法指针            | 未定义行为 |

------

## 4. 写 C 程序的内存管理习惯

### malloc 后检查 NULL

```c
int *p = malloc(sizeof(int));

if (p == NULL) {
    exit(1);
}
```

### 字符串分配记得 +1

```c
char *dst = malloc(strlen(src) + 1);
```

### malloc 后及时初始化

```c
int *p = malloc(sizeof(int));
*p = 0;
```

或者：

```c
int *p = calloc(1, sizeof(int));
```

### 用完就 free

```c
free(p);
```

### free 后设为 NULL

```c
free(p);
p = NULL;
```

### 只 free malloc 返回的原始指针

```c
int *p = malloc(10 * sizeof(int));

free(p);      // 正确
free(p + 1);  // 错误
```

# 最重要的一句话

> C 语言中的堆内存需要程序员自己管理：申请多少、是否初始化、何时释放、释放谁，都必须自己负责。`malloc/free` 是用户态库函数，底层通过 `brk/sbrk/mmap` 向操作系统申请或释放内存资源。
