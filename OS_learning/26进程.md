# 第26章 并发：介绍

## 1. 本章核心主题

本章开始引入操作系统中的新抽象：**线程**。

前面我们已经学过两个重要抽象：

- **虚拟 CPU**：让一个物理 CPU 看起来像多个 CPU，使多个程序好像同时运行。
- **虚拟内存**：让每个进程好像拥有独立、巨大的私有地址空间。

本章引入的是：

> 在一个进程内部创建多个执行流，也就是线程。

线程让一个程序可以同时做多件事情，但也带来了新的问题：**并发安全问题**。

------

## 2. 什么是线程？

线程可以理解为：

> 进程内部的一个执行流。

一个普通进程通常只有一个执行点，也就是一个程序计数器 PC。

多线程程序中，一个进程可以有多个执行点，也就是多个线程。每个线程都有自己的执行状态。

每个线程拥有：

- 程序计数器 PC
- 寄存器
- 栈
- 执行状态

同一进程内的线程共享：

- 地址空间
- 代码段
- 全局变量
- 堆
- 打开的文件

所以线程可以理解为：

> 线程像轻量级进程，但同一进程内的线程共享内存空间。

------

## 3. 线程和进程的区别

| 对比项   | 进程                       | 线程                             |
| -------- | -------------------------- | -------------------------------- |
| 地址空间 | 各自独立                   | 同一进程内共享                   |
| 切换开销 | 较大                       | 较小                             |
| 页表     | 切换进程时通常需要切换页表 | 同一进程内线程切换不需要切换页表 |
| 数据共享 | 需要 IPC                   | 可以直接共享全局变量和堆         |
| 安全性   | 进程之间隔离较强           | 一个线程出错可能影响整个进程     |

一句话总结：

> 进程隔离性更强，线程共享更方便，但也更危险。

------

## 4. 线程控制块 TCB

进程切换时，操作系统会把进程状态保存到 PCB 中。

> PCB：Process Control Block，进程控制块。

线程也需要保存自己的状态，所以操作系统需要维护线程控制块。

> TCB：Thread Control Block，线程控制块。

TCB 中通常保存：

- 线程的程序计数器 PC
- 线程寄存器
- 栈指针
- 线程状态
- 调度相关信息

线程上下文切换时，操作系统会：

```text
保存线程 T1 的 PC、寄存器、栈指针等状态
恢复线程 T2 的 PC、寄存器、栈指针等状态
继续执行线程 T2
```

和进程切换相比，线程切换的重要区别是：

> 同一进程内的线程共享地址空间，因此线程切换时通常不需要切换页表。

------

## 5. 多线程中的栈

单线程进程中，通常只有一个栈。

多线程进程中，每个线程都有自己的栈。

因为每个线程都可能独立调用函数，而函数调用需要使用栈来保存：

- 局部变量
- 函数参数
- 返回地址
- 临时数据

因此，多线程进程的地址空间大致可以理解为：

```text
代码段：所有线程共享
全局变量区：所有线程共享
堆：所有线程共享

线程1栈：线程1私有
线程2栈：线程2私有
线程3栈：线程3私有
```

栈是线程私有的，而堆和全局变量是线程共享的。

# 26.1 线程创建

## 1. pthread_create

线程可以通过 `pthread_create()` 创建。

示例：

```c
pthread_create(&p1, NULL, mythread, "A");
pthread_create(&p2, NULL, mythread, "B");
```

含义是：

```text
创建线程 p1，让它执行 mythread("A")
创建线程 p2，让它执行 mythread("B")
```

新线程创建后，可能马上运行，也可能处于就绪状态，等待调度器安排。

------

## 2. pthread_join

`pthread_join()` 用来等待某个线程结束。

示例：

```c
pthread_join(p1, NULL);
pthread_join(p2, NULL);
```

含义是：

```text
主线程等待 p1 结束
主线程等待 p2 结束
两个线程都结束后，主线程继续执行
```

所以如果代码中有：

```c
pthread_join(p1, NULL);
pthread_join(p2, NULL);
printf("main: end\n");
```

那么：

```text
main: end
```

一定会在两个子线程结束之后打印。

------

## 3. 线程执行顺序不确定

虽然代码中可能先创建线程 1，再创建线程 2：

```c
pthread_create(&p1, NULL, mythread, "A");
pthread_create(&p2, NULL, mythread, "B");
```

但输出不一定是：

```text
A
B
```

也可能是：

```text
B
A
```

原因是：

> 线程什么时候运行由调度器决定，而不是完全由代码书写顺序决定。

因此，多线程程序的执行顺序具有不确定性。

------

# 26.2 共享数据为什么危险

## 1. 共享变量 counter

考虑下面的代码：

```c
static volatile int counter = 0;

void *mythread(void *arg)
{
    for (int i = 0; i < 1e7; i++) {
        counter = counter + 1;
    }
    return NULL;
}
```

如果两个线程都执行这段循环，每个线程都让 `counter` 加 1000 万次。

理论上最终结果应该是：

```text
20000000
```

但是实际运行结果可能是：

```text
19345221
19221041
```

甚至每次运行结果都不一样。

这说明：

> 多个线程同时修改共享变量时，可能会出现错误。

------

## 2. counter++ 不是原子操作

表面上看：

```c
counter = counter + 1;
```

好像是一行代码。

但是在机器底层，它可能会被翻译成三条指令：

```asm
mov counter, %eax
add $1, %eax
mov %eax, counter
```

也就是：

```text
读取 counter 到寄存器
寄存器加 1
把寄存器结果写回 counter
```

问题在于：

> 这三步之间可能发生线程切换。

------

## 3. 更新丢失示例

假设一开始：

```text
counter = 50
```

线程 1 执行：

```text
读取 counter = 50
加 1，得到 51
```

此时线程 1 还没写回，发生线程切换。

线程 2 执行：

```text
读取 counter = 50
加 1，得到 51
写回 counter = 51
```

然后线程 1 恢复执行：

```text
把自己之前算好的 51 写回 counter
```

最终结果是：

```text
counter = 51
```

但正确结果应该是：

```text
counter = 52
```

这就是一次更新被丢失了。

------

# 26.3 核心问题：不可控的调度

并发程序的问题本质是：

> 线程调度顺序不可控。

程序员写代码时看到的是顺序代码，但实际运行时，线程可能在任意时刻被中断和切换。

这会导致：

- 多个线程交错执行
- 共享数据被同时修改
- 程序结果依赖于调度时机
- 每次运行结果可能不同

------

## 1. 竞态条件

竞态条件，英文是 **race condition**。

它指的是：

> 多个线程同时访问共享资源，并且最终结果取决于线程执行的时机和顺序。

例如两个线程同时执行：

```c
counter = counter + 1;
```

如果线程切换发生在不合适的位置，就会导致结果错误。

------

## 2. 不确定性

不确定性，英文是 **indeterminate**。

它指的是：

> 同一个程序，多次运行可能产生不同结果。

例如：

```text
第一次运行：20000000
第二次运行：19345221
第三次运行：19221041
```

这不是 CPU 算错了，而是线程调度顺序不同导致的。

------

## 3. 临界区

临界区，英文是 **critical section**。

它指的是：

> 访问共享变量或共享资源的一段代码。

例如：

```c
counter = counter + 1;
```

这段代码会访问并修改共享变量 `counter`，因此它是临界区。

临界区的问题是：

> 不能让多个线程同时进入，否则可能发生竞态条件。

------

## 4. 互斥

互斥，英文是 **mutual exclusion**。

它的含义是：

> 同一时刻只允许一个线程进入临界区。

例如：

```text
线程 A 正在修改 counter
线程 B 必须等待

线程 A 修改完离开临界区
线程 B 才能进入临界区
```

互斥是后面学习锁的核心目标。

------

# 26.4 原子性愿望

## 1. 什么是原子性？

原子性，英文是 **atomicity**。

它的意思是：

> 一组操作作为一个不可分割的整体执行，要么全部完成，要么完全没有发生，中间状态不能被其他线程看到。

也可以简单理解为：

```text
全部完成，或者完全没做。
不能只做一半。
```

例如，我们希望下面三条指令：

```asm
mov counter, %eax
add $1, %eax
mov %eax, counter
```

能够像一条指令一样执行，中间不能被打断。

------

## 2. 理想的原子指令

书中假设有一条超级指令：

```asm
memory-add 0x8049a1c, $0x1
```

它可以把内存地址中的值直接加 1，并且硬件保证它是原子的。

如果有这条指令，那么：

```c
counter = counter + 1;
```

就不会被线程切换打断，也不会出现更新丢失的问题。

------

## 3. 为什么不能只靠硬件提供所有原子指令？

简单的加一操作可以设计硬件指令支持。

但现实中的临界区可能很复杂，例如：

- 更新链表
- 插入 B 树
- 修改哈希表
- 更新文件系统 inode
- 修改页表
- 分配磁盘块

硬件不可能为每一种复杂操作都设计专门的原子指令。

所以真正的做法是：

> 硬件提供少量基础原子指令，操作系统和程序员在这些基础上构建更高级的同步机制。

这些机制叫做同步原语。

------

## 4. 同步原语

同步原语，英文是 **synchronization primitive**。

它指的是：

> 用来协调多个线程执行顺序的基础工具。

常见同步原语包括：

- 锁 mutex
- 条件变量 condition variable
- 信号量 semaphore

它们主要解决两类问题：

| 问题                                 | 解决方式                  |
| ------------------------------------ | ------------------------- |
| 多个线程不能同时进入临界区           | 使用锁实现互斥            |
| 一个线程需要等待另一个线程完成某件事 | 使用条件变量实现等待/唤醒 |

------

# 26.5 还有一个问题：等待另一个线程

并发中不只有共享变量问题，还有等待问题。

有时候，一个线程必须等待另一个线程完成某件事情后才能继续。

例如：

```text
线程 A 需要使用某个数据
线程 B 负责准备这个数据

数据没准备好时，线程 A 应该睡眠
线程 B 准备好数据后，唤醒线程 A
```

这类问题不是单纯的互斥问题，而是：

> 等待 / 唤醒问题。

后面会通过条件变量来解决。

------

## 1. 锁和条件变量的区别

| 工具                        | 主要解决的问题                               |
| --------------------------- | -------------------------------------------- |
| 锁 mutex                    | 保证同一时刻只有一个线程进入临界区           |
| 条件变量 condition variable | 让一个线程等待某个条件成立，再被其他线程唤醒 |
| 信号量 semaphore            | 可用于互斥，也可用于资源计数和等待/唤醒      |

简单来说：

```text
锁解决“别一起进来”
条件变量解决“条件没满足就先睡，满足后再醒”
```

------

# 26.6 为什么操作系统课要研究并发？

操作系统本身就是典型的并发程序。

操作系统内部有大量共享数据结构，例如：

- 进程表
- 页表
- 调度队列
- 文件系统 inode
- 磁盘块位图
- 缓冲区缓存

这些数据结构可能被多个进程、多个线程或中断处理程序同时访问。

------

## 1. 文件写入示例

假设两个进程同时调用：

```c
write()
```

并且都想向同一个文件末尾追加内容。

操作系统可能需要修改：

- 文件大小
- 文件 inode
- 磁盘块分配信息
- 文件数据块
- 空闲块位图

这些修改必须正确同步。

如果中间发生不合适的中断或并发访问，就可能导致文件系统数据损坏。

所以操作系统必须使用同步原语来保护内部数据结构。

------

## 2. 原子性不仅用于线程

原子性是计算机系统中的重要思想，不只用于多线程。

它还广泛存在于：

- 文件系统
- 数据库事务
- 分布式系统
- 并发数据结构
- 日志系统
- 写时复制技术

例如数据库事务要求：

```text
要么全部提交，要么全部回滚。
```

这就是原子性。

文件系统也类似：

```text
要么一次更新完整完成
要么系统崩溃后恢复到旧状态
不能停在一半新、一半旧的状态
```

# 本章核心概念总结

## 1. 线程

线程是进程内部的执行流。

同一进程内多个线程共享地址空间，但每个线程有自己的 PC、寄存器和栈。

------

## 2. 共享地址空间

线程共享地址空间带来了方便，也带来了危险。

方便之处是：

```text
线程之间可以直接访问共享变量
```

危险之处是：

```text
多个线程同时修改共享变量时可能产生错误
```

------

## 3. 竞态条件

竞态条件是指：

> 程序结果依赖于多个线程执行的具体时机和顺序。

------

## 4. 临界区

临界区是：

> 访问共享资源的一段代码。

临界区必须被保护，不能让多个线程同时执行。

------

## 5. 互斥

互斥是：

> 保证同一时刻只有一个线程进入临界区。

------

## 6. 原子性

原子性是：

> 一组操作要么全部完成，要么完全不发生，中间状态不可见。

------

## 7. 同步原语

同步原语是协调多个线程执行的基础工具。

常见同步原语包括：

- 锁
- 条件变量
- 信号量

# 考试重点

## 1. 什么是线程？

线程是进程内部的一个执行流，拥有自己的 PC、寄存器和栈，但与同一进程内的其他线程共享地址空间。

------

## 2. 线程和进程有什么区别？

进程拥有独立地址空间，线程共享同一进程的地址空间。线程切换开销更小，但共享数据会带来并发安全问题。

------

## 3. 为什么 `counter++` 不是线程安全的？

因为 `counter++` 不是一个原子操作，它通常包含读取、加一、写回三个步骤。线程可能在这几个步骤之间发生切换，导致更新丢失。

------

## 4. 什么是竞态条件？

多个线程同时访问共享资源，程序结果取决于线程执行的时机和顺序，这种情况叫竞态条件。

------

## 5. 什么是临界区？

临界区是访问共享变量或共享资源的一段代码。多个线程不能同时进入临界区。

------

## 6. 什么是互斥？

互斥是指同一时刻只允许一个线程进入临界区，从而避免多个线程同时修改共享数据。

------

## 7. 什么是原子性？

原子性是指一组操作作为一个不可分割的整体执行，要么全部完成，要么完全不执行，中间状态不能被其他线程看到。

------

## 8. 为什么不能只靠硬件提供复杂原子指令？

因为程序中的临界区形式太多，可能涉及复杂数据结构，例如链表、B 树、文件系统结构等，硬件不可能为每种操作都设计专门指令。

------

## 9. 什么是同步原语？

同步原语是用来协调多个线程执行顺序的基础工具，例如锁、条件变量和信号量。

------

## 10. 为什么操作系统课要学习并发？

因为操作系统本身就是并发程序，内部有大量共享数据结构，例如进程表、页表、文件系统结构等。操作系统必须使用同步机制保证这些数据结构被正确访问。

------

# 一句话总结

> 线程让一个进程内部可以同时拥有多个执行流，但由于线程共享地址空间，多个线程同时访问共享数据时可能产生竞态条件。因此，我们需要互斥、原子性和同步原语来保证并发程序的正确性。
