# 第 6 章 机制：受限直接执行

## 1. 本章核心问题

操作系统要虚拟化 CPU，让多个进程共享一个或多个 CPU，看起来像是同时运行。

核心矛盾：

```text
既要高性能：让程序直接在 CPU 上运行
又要可控制：操作系统必须能限制程序、重新拿回 CPU
```

因此提出机制：

> **受限直接执行 Limited Direct Execution，LDE**

------

## 2. 什么是受限直接执行

受限直接执行可以拆成两部分：

```text
直接执行：程序直接在 CPU 上运行，保证性能
受限：程序权限受到限制，不能随便操作硬件和系统资源
```

一句话总结：

> 让用户程序直接运行以获得高性能，同时通过硬件和操作系统机制限制它，保证操作系统始终掌握控制权。

------

## 3. CPU 虚拟化的基本思想

操作系统通过 **时分共享 time sharing** 实现 CPU 虚拟化。

```text
运行进程 A 一小段时间
↓
切换到进程 B
↓
再切换到进程 C
↓
再切回进程 A
```

由于切换速度很快，用户感觉多个程序像是在同时运行。

------

# 6.1 基本技巧：受限直接执行

## 1. 直接执行的基本流程

如果没有任何限制，操作系统运行程序的大致流程是：

```text
操作系统：
1. 创建进程表项
2. 分配内存
3. 将程序加载到内存
4. 设置 argc/argv 和用户栈
5. 清空寄存器
6. 跳转到 main()

用户程序：
7. 执行 main()
8. main() 返回

操作系统：
9. 回收内存
10. 从进程列表中删除该进程
```

## 2. 直接执行的问题

直接执行虽然快，但有两个大问题：

### 问题一：程序做危险操作怎么办？

例如用户程序想：

```text
直接访问磁盘
修改别的进程内存
关闭中断
修改页表
操作硬件设备
```

如果不限制，系统安全性会崩掉。

### 问题二：程序不主动停止怎么办？

例如：

```c
while (1) {
}
```

如果程序死循环，操作系统如何重新拿回 CPU？

------

# 6.2 问题 1：受限制的操作

## 1. 用户态和内核态

硬件提供两种执行模式：

## 用户模式 user mode

普通应用程序运行在用户模式。

在用户模式下，程序不能直接执行特权操作。

例如不能：

```text
直接访问磁盘
直接访问网卡
修改页表
关闭中断
设置陷阱表
```

如果强行执行，CPU 会触发异常，操作系统可能终止该进程。

## 内核模式 kernel mode

操作系统运行在内核模式。

在内核模式下，代码拥有最高权限，可以执行特权操作。

例如：

```text
读写磁盘
管理内存
创建进程
处理中断
修改页表
访问设备
```

------

## 2. 系统调用 system call

用户程序不能直接做危险操作，但它仍然需要读文件、写文件、创建进程等功能。

解决方法是：

> 用户程序通过系统调用请求操作系统帮它完成受限操作。

例如：

```c
read(fd, buf, 100);
open("a.txt", O_RDONLY);
write(fd, buf, 100);
fork();
execve();
exit();
```

这些看起来像普通函数调用，但内部会进入内核。

------

## 3. trap：陷入内核

系统调用的关键机制是：

```text
trap
```

trap 是一种受保护的控制权转移。

系统调用过程：

```text
用户程序调用 read/open 等库函数
↓
C 库准备参数和系统调用号
↓
执行 trap 指令
↓
CPU 从用户模式切换到内核模式
↓
跳转到操作系统的系统调用处理程序
↓
操作系统执行服务
↓
return-from-trap
↓
CPU 切回用户模式
↓
用户程序继续执行
```

------

## 4. 为什么系统调用看起来像函数调用？

因为 `open()`、`read()` 等通常是 C 库函数。

它们内部会：

```text
准备系统调用号
准备参数
执行 trap/syscall 指令
处理返回值
返回给用户程序
```

所以表面上像普通函数调用，本质上是进入操作系统内核。

------

## 5. 陷阱表 trap table

CPU 发生系统调用、中断或异常后，需要知道应该跳到内核的哪段代码。

这个映射由陷阱表保存。

```text
系统调用    → 系统调用处理程序
时钟中断    → 时钟中断处理程序
键盘中断    → 键盘中断处理程序
非法指令    → 非法指令异常处理程序
缺页异常    → 缺页异常处理程序
```

陷阱表由操作系统在启动时设置。

用户程序不能设置陷阱表，否则它可以控制内核入口，甚至接管系统。

------

## 6. 内核栈 kernel stack

当用户程序通过 trap 进入内核时，硬件需要保存当前程序状态。

通常保存：

```text
程序计数器 PC
寄存器
标志位
栈指针
返回地址
```

这些信息保存到该进程的内核栈中。

目的：

> 系统调用结束后，能够恢复现场，继续执行用户程序。

------

# 6.3 问题 2：在进程之间切换

## 1. 关键问题

如果一个进程正在 CPU 上运行，那么操作系统没有运行。

问题是：

> 操作系统没有运行时，怎么重新拿回 CPU？

------

## 2. 协作式方式 cooperative

早期系统可能依赖进程主动交出 CPU。

进程通过以下方式交出 CPU：

```text
系统调用
yield()
非法操作导致异常
```

例如：

```c
yield();
read(fd, buf, 100);
exit();
```

这种方式的问题是：

```c
while (1) {
}
```

如果进程死循环，且不系统调用、不 yield、不出错，那么操作系统可能永远拿不回 CPU。

所以协作式方式不可靠。

------

## 3. 非协作方式：时钟中断 timer interrupt

现代操作系统依靠硬件时钟中断重新获得控制权。

流程：

```text
进程 A 正在运行
↓
时钟中断发生
↓
CPU 暂停进程 A
↓
保存 A 的用户态上下文
↓
切换到内核模式
↓
执行时钟中断处理程序
↓
操作系统重新获得 CPU
```

时钟中断的作用：

> 即使进程不主动交出 CPU，操作系统也能定期重新获得控制权。

可以记成：

```text
系统调用：进程主动进入内核
时钟中断：硬件强制进入内核
```

------

## 4. 调度程序 scheduler

操作系统重新获得 CPU 后，需要决定：

```text
继续运行当前进程？
还是切换到另一个进程？
```

这个决定由调度程序完成。

调度策略是后面章节的重点。

------

## 5. 上下文切换 context switch

如果调度程序决定切换进程，操作系统就执行上下文切换。

上下文切换本质：

```text
保存当前进程的上下文
恢复下一个进程的上下文
```

上下文包括：

```text
程序计数器 PC
通用寄存器
栈指针
内核栈指针
进程状态
```

------

## 6. A 进程切换到 B 进程的过程

```text
进程 A 在用户态运行
↓
时钟中断发生
↓
硬件保存 A 的用户态上下文到 A 的内核栈
↓
CPU 切换到内核态
↓
进入时钟中断处理程序
↓
调度程序决定切换到 B
↓
OS 保存 A 的内核上下文
↓
OS 恢复 B 的内核上下文
↓
切换到 B 的内核栈
↓
return-from-trap
↓
恢复 B 的用户态上下文
↓
CPU 切回用户态
↓
进程 B 开始运行
```

关键点：

> `return-from-trap` 最后不是返回到 A，而是返回到 B。

------

## 7. 两种保存/恢复

本节有两类保存和恢复，要区分清楚。

### 第一类：硬件保存用户态上下文

发生时钟中断或 trap 时，硬件自动保存当前进程的用户态寄存器。

通常保存到：

```text
当前进程的内核栈
```

### 第二类：操作系统保存内核态上下文

当 OS 决定从 A 切换到 B 时，OS 执行上下文切换。

它会：

```text
保存 A 的内核上下文到 A 的进程结构
从 B 的进程结构恢复 B 的内核上下文
切换到 B 的内核栈
```

简记：

```text
中断发生：硬件保存用户现场
进程切换：OS 保存内核现场
```

------

# 6.4 担心并发吗

## 1. 内核中的并发问题

可能出现这种情况：

```text
内核正在处理系统调用
↓
时钟中断又发生了
```

或者：

```text
内核正在处理一个中断
↓
另一个中断又来了
```

这会带来并发问题。

因为内核中有很多共享数据结构：

```text
进程表
就绪队列
页表
文件表
缓冲区
设备队列
```

如果多个内核活动同时修改这些结构，可能发生错误。

------

## 2. 禁用中断 disable interrupt

一种简单方法是：

```text
处理中断或关键代码时临时禁用中断
```

流程：

```text
关闭中断
↓
修改关键数据结构
↓
打开中断
```

优点：

```text
简单直接
防止当前 CPU 被中断打断
```

缺点：

```text
不能禁用太久
否则可能导致响应慢、丢中断、系统卡顿
```

------

## 3. 加锁 locking

另一种方式是使用锁保护共享数据结构。

例如：

```text
访问进程表前先加锁
修改完成后释放锁
```

这在多处理器系统中特别重要。

后续并发章节会详细讨论：

```text
锁
信号量
条件变量
死锁
竞态条件
多处理器同步
```

# 考试背诵版

为了虚拟化 CPU，操作系统采用受限直接执行机制。用户程序大多数时间直接在 CPU 上运行，以保证性能。为了防止用户程序执行危险操作，硬件提供用户态和内核态，用户程序需要执行 I/O 等特权操作时，必须通过系统调用执行 trap 进入内核。操作系统在启动时设置陷阱表，并启动时钟中断。时钟中断保证即使进程不主动让出 CPU，操作系统也能重新获得控制权。操作系统获得控制权后，调度程序决定是否切换进程。如果需要切换，则通过上下文切换保存当前进程状态、恢复另一个进程状态。为了处理内核中的并发问题，操作系统还会使用禁用中断和加锁等机制保护共享数据结构。

------

# 一句话总结

> 受限直接执行就是：让用户程序直接在 CPU 上运行以获得高性能，同时通过用户态/内核态、系统调用、trap、时钟中断和上下文切换，让操作系统始终保持对 CPU 和系统资源的控制。
