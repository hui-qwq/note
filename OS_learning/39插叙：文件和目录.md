# 第39章 插叙：文件和目录

## 一、本章核心

前面操作系统已经虚拟化了两类资源：

| 抽象     | 虚拟化对象 |
| -------- | ---------- |
| 进程     | CPU        |
| 地址空间 | 内存       |
| 文件系统 | 持久存储   |

本章开始讨论第三类重要资源：**持久存储**。

持久存储设备包括：

- 硬盘 HDD
- 固态硬盘 SSD
- U 盘
- 光盘
- 其他长期保存数据的设备

内存断电后数据会丢失，而持久存储设备断电后数据仍然存在。因此，文件系统必须保证用户数据能够被可靠保存和再次读取。

本章的核心问题是：

> 操作系统如何管理持久存储设备？
> 文件系统提供哪些 API？
> 文件、目录、inode、链接、挂载之间是什么关系？

------

# 二、文件和目录

## 1. 文件 File

文件是文件系统提供的第一个核心抽象。

文件可以理解为：

> 一个线性的字节数组。

文件系统通常不关心文件内部到底是什么。

它可能是：

- 文本文件
- C 源代码
- 图片
- 音乐
- 视频
- 可执行程序

但对文件系统来说，它们本质上都是一串字节。

文件系统的责任是：

1. 把数据持久保存到磁盘上；
2. 以后读取时，返回原来写入的数据；
3. 管理文件的名字、大小、权限、位置等元数据。

------

## 2. inode：文件的低级名字

每个文件都有一个底层编号，称为：

> inode number，简称 inode 号。

用户看到的是文件名，例如：

```bash
foo.txt
main.c
hello.jpg
```

但文件系统内部真正识别文件，靠的是 inode 号。

可以简单理解为：

```text
文件名 → inode号 → 文件数据
```

例如：

```text
foo → inode 10
```

文件名是给人看的，inode 才是文件系统内部定位文件的关键。

------

## 3. 目录 Directory

目录是文件系统提供的第二个核心抽象。

目录本身也有 inode 号，但目录的内容很特殊。

目录保存的是一组映射关系：

```text
用户可读名字 → inode号
```

例如：

```text
("foo", 10)
("bar.txt", 20)
```

也就是：

| 文件名/目录名 | inode号 |
| ------------- | ------- |
| foo           | 10      |
| bar.txt       | 20      |

因此，目录本质上就是一张“名字到 inode 的映射表”。

------

## 4. 目录树

目录可以包含文件，也可以包含其他目录。

这样就形成了目录树。

UNIX 系统的目录树从根目录开始：

```bash
/
```

例如路径：

```bash
/foo/bar.txt
```

含义是：

1. 从根目录 `/` 开始；
2. 进入 `foo` 目录；
3. 找到 `bar.txt` 文件。

------

## 5. 绝对路径和相对路径

### 绝对路径

从根目录 `/` 开始的路径，叫绝对路径。

例如：

```bash
/foo/bar.txt
/bar/foo/bar.txt
```

特点：

- 从 `/` 开始；
- 不依赖当前目录；
- 可以唯一定位一个文件或目录。

### 相对路径

不从 `/` 开始，而是相对于当前所在目录的路径。

例如当前目录是 `/foo`，那么：

```bash
bar.txt
```

表示：

```bash
/foo/bar.txt
```

------

## 6. 文件后缀只是约定

文件名常带有后缀，例如：

| 后缀   | 常见含义 |
| ------ | -------- |
| `.c`   | C 源代码 |
| `.txt` | 文本文件 |
| `.jpg` | 图片     |
| `.mp3` | 音频     |

但文件后缀通常只是约定，并不由文件系统强制保证。

也就是说，一个叫 `main.c` 的文件，里面不一定真的是 C 代码。

------

# 三、基础文件系统接口

## 1. 常见系统调用

| 功能           | 系统调用             |
| -------------- | -------------------- |
| 创建/打开文件  | `open()`             |
| 读取文件       | `read()`             |
| 写入文件       | `write()`            |
| 关闭文件       | `close()`            |
| 修改文件偏移量 | `lseek()`            |
| 强制写入磁盘   | `fsync()`            |
| 重命名文件     | `rename()`           |
| 获取文件信息   | `stat()` / `fstat()` |
| 删除文件名链接 | `unlink()`           |
| 创建目录       | `mkdir()`            |
| 删除目录       | `rmdir()`            |

------

## 2. 创建文件：open()

创建文件可以使用 `open()` 系统调用。

```c
int fd = open("foo", O_CREAT | O_WRONLY | O_TRUNC);
```

含义：

| 标志       | 作用                         |
| ---------- | ---------------------------- |
| `O_CREAT`  | 如果文件不存在，就创建它     |
| `O_WRONLY` | 以只写方式打开               |
| `O_TRUNC`  | 如果文件已存在，就清空原内容 |

所以这行代码表示：

> 创建或打开当前目录下的 `foo` 文件，只用于写入；如果它已经存在，就先清空它。

------

## 3. creat() 系统调用

早期 UNIX 中也可以用：

```c
int fd = creat("foo");
```

它基本等价于：

```c
open("foo", O_CREAT | O_WRONLY | O_TRUNC);
```

现在更常用 `open()`，因为它更灵活。

------

# 四、文件描述符

## 1. 什么是文件描述符

`open()` 的返回值是一个整数，叫：

> 文件描述符，file descriptor，简称 fd。

例如：

```c
int fd = open("foo", O_RDONLY);
```

如果成功，可能返回：

```text
fd = 3
```

之后读写文件都通过这个 fd 进行：

```c
read(fd, buffer, size);
write(fd, buffer, size);
close(fd);
```

文件描述符可以理解为：

> 当前进程访问某个文件的句柄。

------

## 2. 文件描述符是进程私有的

文件描述符是每个进程私有的。

例如：

```text
进程 A 的 fd=3
进程 B 的 fd=3
```

它们不一定指向同一个文件。

------

## 3. 默认打开的三个文件描述符

每个 UNIX 进程启动时，默认已经打开了三个文件描述符：

| fd   | 名称            | 含义               |
| ---- | --------------- | ------------------ |
| 0    | 标准输入 stdin  | 从键盘或输入源读取 |
| 1    | 标准输出 stdout | 输出到屏幕         |
| 2    | 标准错误 stderr | 输出错误信息       |

所以第一个由程序自己打开的普通文件，通常会得到 fd=3。

------

# 五、读写文件

## 1. read()

读取文件使用：

```c
ssize_t read(int fd, void *buf, size_t count);
```

参数含义：

| 参数    | 含义                 |
| ------- | -------------------- |
| `fd`    | 文件描述符           |
| `buf`   | 存放读取结果的缓冲区 |
| `count` | 最多读取多少字节     |

返回值含义：

| 返回值 | 含义             |
| ------ | ---------------- |
| `> 0`  | 实际读取的字节数 |
| `0`    | 文件读完，EOF    |
| `< 0`  | 读取出错         |

------

## 2. write()

写入文件使用：

```c
ssize_t write(int fd, const void *buf, size_t count);
```

例如：

```c
write(1, "hello\n", 6);
```

其中 `fd=1` 表示标准输出，因此这行代码会把 `hello` 输出到屏幕。

------

## 3. close()

文件使用完后，需要关闭：

```c
close(fd);
```

关闭后，该 fd 不再指向原文件，操作系统可以回收相关资源。

------

## 4. cat 命令背后的系统调用

假设执行：

```bash
echo hello > foo
cat foo
```

使用 `strace cat foo` 可以看到类似过程：

```text
open("foo", O_RDONLY) = 3
read(3, "hello\n", 4096) = 6
write(1, "hello\n", 6) = 6
read(3, "", 4096) = 0
close(3) = 0
```

解释：

1. `open()` 打开文件，返回 fd=3；
2. `read()` 从文件中读出 `hello\n`；
3. `write()` 写到标准输出 fd=1；
4. 再次 `read()` 返回 0，表示 EOF；
5. `close()` 关闭文件。

------

# 六、不按顺序读写：lseek()

## 1. 顺序访问和随机访问

默认情况下，`read()` 和 `write()` 是顺序访问。

也就是：

```text
从当前偏移量开始读/写
读/写多少字节，当前偏移量就向后移动多少字节
```

如果想跳到文件中的某个位置读写，就需要：

```c
off_t lseek(int fildes, off_t offset, int whence);
```

------

## 2. lseek() 的参数

```c
lseek(fd, offset, whence);
```

| 参数     | 含义               |
| -------- | ------------------ |
| `fd`     | 文件描述符         |
| `offset` | 偏移量             |
| `whence` | 从哪里开始计算偏移 |

`whence` 常见取值：

| whence     | 含义               |
| ---------- | ------------------ |
| `SEEK_SET` | 从文件开头开始算   |
| `SEEK_CUR` | 从当前偏移量开始算 |
| `SEEK_END` | 从文件末尾开始算   |

------

## 3. lseek() 不会真的执行磁盘寻道

虽然名字叫 `lseek()`，但它不会立刻移动磁盘磁头，也不会马上执行磁盘 I/O。

它只是修改内核中的一个变量：

```text
当前文件偏移量
```

真正的 I/O 发生在后面的：

```c
read()
write()
```

------

# 七、立即写入磁盘：fsync()

## 1. write() 不保证立即落盘

普通的：

```c
write(fd, buffer, size);
```

通常只是把数据写入内存缓存。

文件系统可能过几秒或几十秒后，再统一把数据写入磁盘。

好处是性能更高。

风险是：

> 如果 write() 返回后，数据还没落盘，机器突然崩溃，数据可能丢失。

------

## 2. fsync() 的作用

如果必须保证数据已经写入磁盘，需要调用：

```c
fsync(fd);
```

它会强制把该文件的脏数据写入磁盘。

脏数据指的是：

```text
已经在内存中修改，但还没写入磁盘的数据
```

------

## 3. fsync() 示例

```c
int fd = open("foo", O_CREAT | O_WRONLY | O_TRUNC);
assert(fd > -1);

int rc = write(fd, buffer, size);
assert(rc == size);

rc = fsync(fd);
assert(rc == 0);

close(fd);
```

执行完 `fsync(fd)` 后，可以认为文件数据已经被强制写入磁盘。

------

## 4. 有时还需要 fsync 目录

如果是新创建的文件，只 `fsync(fd)` 有时还不够。

因为文件内容可能已经落盘，但目录项：

```text
文件名 → inode
```

可能还没有落盘。

严格来说，有些场景还需要对包含该文件的目录执行 `fsync()`，这样才能保证文件名也持久化。

------

# 八、文件重命名：rename()

## 1. rename() 的使用

命令行中：

```bash
mv foo bar
```

底层通常使用：

```c
rename("foo", "bar");
```

含义是把旧名字 `foo` 改成新名字 `bar`。

------

## 2. rename() 的原子性

`rename()` 通常提供原子性。

也就是说，即使系统在重命名过程中崩溃，文件也只会处于两种状态之一：

```text
要么还是旧名字
要么已经是新名字
```

不会出现中间状态。

------

## 3. 原子文件更新

很多编辑器保存文件时，会采用如下流程：

```c
int fd = open("foo.txt.tmp", O_WRONLY | O_CREAT | O_TRUNC);

write(fd, buffer, size);

fsync(fd);

close(fd);

rename("foo.txt.tmp", "foo.txt");
```

这叫原子文件更新。

好处是：

| 崩溃位置         | 结果                   |
| ---------------- | ---------------------- |
| 写临时文件时崩溃 | 原文件仍然存在         |
| fsync 前崩溃     | 原文件仍然存在         |
| rename 后崩溃    | 新文件生效             |
| rename 中崩溃    | 要么旧文件，要么新文件 |

这样可以避免文件保存到一半损坏。

------

# 九、获取文件信息：stat() 和 fstat()

## 1. 文件元数据

文件系统除了保存文件内容，还保存文件元数据。

元数据包括：

- 文件大小
- inode 号
- 权限
- 拥有者
- 所属组
- 链接数
- 访问时间
- 修改时间
- 状态改变时间
- 占用块数

------

## 2. stat() 和 fstat()

获取元数据可以使用：

| 调用      | 说明                       |
| --------- | -------------------------- |
| `stat()`  | 通过路径名获取文件信息     |
| `fstat()` | 通过文件描述符获取文件信息 |

示例：

```c
struct stat s;
stat("foo", &s);
```

或者：

```c
struct stat s;
fstat(fd, &s);
```

------

## 3. inode 保存元数据

文件系统通常将文件元数据保存在 inode 中。

可以理解为：

```text
inode = 文件的元数据结构
```

目录保存：

```text
文件名 → inode号
```

inode 保存：

```text
文件大小、权限、所有者、链接数、数据块位置等
```

------

# 十、删除文件：unlink()

## 1. rm 的底层调用

命令行删除文件：

```bash
rm foo
```

底层通常调用：

```c
unlink("foo");
```

为什么叫 `unlink()`，而不是 `delete()`？

因为文件名和文件本体不是一回事。

------

## 2. unlink() 的真正含义

目录保存的是：

```text
文件名 → inode号
```

例如：

```text
foo → inode 10
```

`unlink("foo")` 的含义是：

> 删除目录中 `foo` 这个名字和 inode 10 之间的链接。

也就是删除目录项，而不是立刻删除 inode 和数据块。

------

## 3. 文件真正删除的条件

一个文件真正被删除，需要满足：

```text
链接数为 0
并且
没有进程正在打开它
```

如果还有文件名指向它，不能回收数据块。

如果没有文件名了，但某个进程还打开着它，也不能马上回收。

------

# 十一、目录操作

## 1. 创建目录：mkdir()

创建目录使用：

```c
mkdir("foo", 0777);
```

命令行中：

```bash
mkdir foo
```

------

## 2. 目录不能直接写入

用户程序不能像普通文件那样直接 `write()` 一个目录。

因为目录内容属于文件系统元数据。

如果允许用户随便写目录，很容易破坏文件系统结构。

目录只能通过系统调用间接修改，例如：

| 操作                      | 影响           |
| ------------------------- | -------------- |
| `mkdir()`                 | 创建目录项     |
| `open(..., O_CREAT, ...)` | 创建文件目录项 |
| `unlink()`                | 删除文件目录项 |
| `rmdir()`                 | 删除目录项     |
| `rename()`                | 修改目录项     |

------

## 3. 空目录中的 . 和 ..

一个刚创建的目录看起来是空的，但实际至少包含两个特殊目录项：

| 名称 | 含义     |
| ---- | -------- |
| `.`  | 当前目录 |
| `..` | 父目录   |

例如：

```bash
mkdir foo
cd foo
ls -a
```

会看到：

```text
.
..
```

------

## 4. 读取目录

读取目录可以使用：

```c
opendir()
readdir()
closedir()
```

示例：

```c
#include <stdio.h>
#include <dirent.h>
#include <assert.h>

int main() {
    DIR *dp = opendir(".");
    assert(dp != NULL);

    struct dirent *d;

    while ((d = readdir(dp)) != NULL) {
        printf("%d %s\n", (int)d->d_ino, d->d_name);
    }

    closedir(dp);
    return 0;
}
```

这个程序类似简化版 `ls`。

------

## 5. struct dirent

目录项结构体大致如下：

```c
struct dirent {
    char d_name[256];       // 文件名
    ino_t d_ino;            // inode号
    off_t d_off;            // 下一个目录项的偏移
    unsigned short d_reclen;// 目录项长度
    unsigned char d_type;   // 文件类型
};
```

最重要的是：

| 字段     | 含义     |
| -------- | -------- |
| `d_name` | 文件名   |
| `d_ino`  | inode 号 |

所以读取目录，本质上就是读取：

```text
文件名 → inode号
```

------

## 6. 为什么 ls -l 更慢

普通 `ls` 主要读取目录项：

```text
文件名 + inode号
```

而：

```bash
ls -l
```

还要显示：

- 权限
- 文件大小
- 所有者
- 所属组
- 修改时间

这些信息在 inode 中。

所以 `ls -l` 需要对每个文件额外调用 `stat()`，因此比普通 `ls` 做更多工作。

------

## 7. 删除目录：rmdir()

删除空目录使用：

```c
rmdir("foo");
```

命令行：

```bash
rmdir foo
```

`rmdir()` 要求目录必须为空。

这里的“空”指的是：

```text
除了 . 和 .. 之外，没有其他目录项
```

如果目录中还有文件或子目录，`rmdir()` 会失败。

------

# 十二、硬链接 Hard Link

## 1. 什么是硬链接

硬链接就是创建一个新的文件名，让它指向已有文件的同一个 inode。

命令：

```bash
ln file file2
```

执行后：

```text
file  → inode 67158084
file2 → inode 67158084
```

两个文件名指向同一个 inode。

------

## 2. 硬链接不会复制文件

硬链接不会复制文件内容。

它只是让目录中多了一个名字，指向同一个 inode。

可以用：

```bash
ls -i file file2
```

查看 inode 号。

如果两个文件 inode 号相同，就说明它们是同一个文件的两个名字。

------

## 3. 链接计数

inode 中有一个链接计数，也叫引用计数。

它表示：

> 当前有多少个文件名指向这个 inode。

例如：

```bash
echo hello > file
```

此时：

```text
file → inode 67158084
Links = 1
```

创建硬链接：

```bash
ln file file2
```

之后：

```text
file  → inode 67158084
file2 → inode 67158084
Links = 2
```

------

## 4. 删除硬链接

假设：

```text
file  → inode 100
file2 → inode 100
file3 → inode 100
```

链接数是 3。

执行：

```bash
rm file
```

只是删除：

```text
file → inode 100
```

剩下：

```text
file2 → inode 100
file3 → inode 100
```

链接数变成 2。

只有当最后一个名字也被删除，链接数变成 0 时，文件系统才会真正回收 inode 和数据块。

------

## 5. 为什么 unlink 这个名字合理

因为删除文件本质上不是直接删除文件内容，而是：

```text
解除 文件名 → inode 的链接
```

所以系统调用叫：

```c
unlink()
```

这个名字非常准确。

------

# 十三、符号链接 Symbolic Link

## 1. 什么是符号链接

符号链接也叫软链接。

创建软链接：

```bash
ln -s file file2
```

它看起来也可以通过 `file2` 访问 `file` 的内容。

------

## 2. 软链接和硬链接完全不同

硬链接：

```text
file  → inode 100
file2 → inode 100
```

软链接：

```text
file  → inode 100

file2 → inode 200
inode 200 的内容是 "file"
```

软链接本身是一个独立文件，它保存的是目标路径字符串。

------

## 3. 查看软链接

执行：

```bash
ls -al
```

可能看到：

```text
-rw-r----- 1 remzi remzi 6 file
lrwxrwxrwx 1 remzi remzi 4 file2 -> file
```

第一列开头表示文件类型：

| 字符 | 类型     |
| ---- | -------- |
| `-`  | 普通文件 |
| `d`  | 目录     |
| `l`  | 符号链接 |

`file2 -> file` 表示 `file2` 是指向 `file` 的软链接。

------

## 4. 软链接大小

软链接的内容是目标路径字符串。

例如：

```bash
ln -s file file2
```

`file2` 保存的是字符串：

```text
file
```

长度是 4，所以软链接大小可能是 4 字节。

如果目标路径更长，软链接文件也更大。

------

## 5. 悬空引用

软链接可能产生悬空引用。

例如：

```bash
echo hello > file
ln -s file file2
cat file2
```

可以正常输出：

```text
hello
```

但如果执行：

```bash
rm file
cat file2
```

会报错：

```text
No such file or directory
```

因为 `file2` 保存的是路径 `"file"`，但 `file` 已经不存在了。

这种情况叫：

> 悬空引用 dangling reference。

------

## 6. 硬链接和软链接对比

| 对比项               | 硬链接                     | 软链接                   |
| -------------------- | -------------------------- | ------------------------ |
| 本质                 | 多个文件名指向同一个 inode | 独立文件，内容是目标路径 |
| 是否有独立 inode     | 没有，和目标共享 inode     | 有                       |
| 是否复制数据         | 不复制                     | 不复制                   |
| 是否增加目标链接计数 | 会                         | 不会                     |
| 是否能跨文件系统     | 通常不能                   | 可以                     |
| 是否能链接目录       | 通常不能                   | 可以                     |
| 删除原文件后         | 其他硬链接仍可用           | 可能变成悬空链接         |
| 创建命令             | `ln old new`               | `ln -s old new`          |

------

# 十四、创建和挂载文件系统

## 1. 创建文件系统：mkfs

一个磁盘分区最开始只是原始空间，不能直接当文件系统使用。

需要先创建文件系统。

常用工具：

```bash
mkfs
```

例如：

```bash
mkfs -t ext3 /dev/sda1
```

含义是：

> 在 `/dev/sda1` 这个设备分区上创建 ext3 文件系统。

------

## 2. 挂载文件系统：mount

创建好文件系统后，还需要把它挂到统一的目录树中。

命令：

```bash
mount -t ext3 /dev/sda1 /home/users
```

含义是：

> 把 `/dev/sda1` 上的 ext3 文件系统挂载到 `/home/users` 目录上。

其中：

```text
/home/users
```

叫挂载点。

------

## 3. 挂载的直观理解

假设 `/dev/sda1` 文件系统内部有：

```text
/
├── a
│   └── foo
└── b
    └── foo
```

挂载到 `/home/users` 后：

```bash
mount -t ext3 /dev/sda1 /home/users
```

就可以通过：

```bash
/home/users/a/foo
/home/users/b/foo
```

访问其中的文件。

------

## 4. mount 的核心作用

`mount` 的作用是：

> 把多个不同设备、不同类型的文件系统，统一接到一棵目录树中。

用户看到的是一棵统一的树：

```text
/
├── home
├── tmp
├── proc
├── sys
├── dev
└── ...
```

但背后可能来自很多不同的文件系统。

------

## 5. 查看当前挂载情况

执行：

```bash
mount
```

可能看到：

```text
/dev/sda1 on / type ext3
proc on /proc type proc
sysfs on /sys type sysfs
tmpfs on /dev/shm type tmpfs
AFS on /afs type afs
```

------

## 6. 常见文件系统类型

| 文件系统    | 说明                   |
| ----------- | ---------------------- |
| `ext3/ext4` | Linux 常见磁盘文件系统 |
| `proc`      | 查看进程和内核信息     |
| `sysfs`     | 暴露设备和内核对象信息 |
| `tmpfs`     | 基于内存的临时文件系统 |
| `AFS`       | 分布式文件系统         |

------

# 十五、本章核心模型

## 1. 文件访问模型

```text
路径名
  ↓
目录查找
  ↓
文件名 → inode号
  ↓
inode
  ↓
文件元数据 + 数据块位置
  ↓
文件内容
```

------

## 2. 普通文件模型

```text
文件名 → inode → 数据块
```

------

## 3. 硬链接模型

```text
file  → inode 100
file2 → inode 100
file3 → inode 100
```

多个名字指向同一个 inode。

------

## 4. 软链接模型

```text
file  → inode 100

file2 → inode 200
inode 200 的内容是 "file"
```

软链接本身是一个独立文件，保存目标路径。

------

## 5. 删除文件模型

```text
unlink("foo")
  ↓
删除目录项 foo → inode
  ↓
链接数减 1
  ↓
如果链接数为 0 且无人打开
  ↓
回收 inode 和数据块
```

------

## 6. 挂载模型

```text
独立文件系统：
/
├── a
└── b

挂载到 /home/users 后：

/home/users
├── a
└── b
```

------

# 十六、容易考的点

## 1. 文件是什么？

文件是线性的字节数组。

------

## 2. 目录是什么？

目录是名字到 inode 号的映射表。

```text
文件名 → inode号
```

------

## 3. inode 是什么？

inode 是文件系统保存文件元数据的数据结构，包含：

- 文件大小
- 权限
- 所有者
- 链接数
- 时间戳
- 数据块位置

------

## 4. 为什么第一个打开的文件通常是 fd=3？

因为每个进程默认已经打开：

| fd   | 含义     |
| ---- | -------- |
| 0    | 标准输入 |
| 1    | 标准输出 |
| 2    | 标准错误 |

所以用户打开的第一个普通文件通常是 3。

------

## 5. read() 返回 0 表示什么？

表示文件已经读完，即 EOF。

------

## 6. lseek() 会不会执行磁盘寻道？

不会。

`lseek()` 只是修改内核中的当前文件偏移量，不会立刻发生磁盘 I/O。

------

## 7. write() 返回成功是否表示数据已经写入磁盘？

不一定。

`write()` 可能只是把数据写入内存缓存。

如果要强制落盘，需要调用：

```c
fsync(fd);
```

------

## 8. rename() 为什么重要？

因为 `rename()` 通常是原子的。

可以用它实现安全的文件替换：

```text
写临时文件 → fsync → close → rename
```

------

## 9. 为什么删除文件叫 unlink？

因为删除文件本质是删除目录中的链接：

```text
文件名 → inode
```

不是立即删除文件内容。

------

## 10. 硬链接和软链接区别？

硬链接是多个文件名指向同一个 inode。

软链接是一个独立文件，里面保存目标路径。

------

## 11. 删除原文件后，硬链接还能用吗？

能。

因为硬链接之间地位平等，只要还有一个名字指向 inode，文件就存在。

------

## 12. 删除原文件后，软链接还能用吗？

通常不能。

因为软链接保存的是路径名，目标路径不存在后，它就变成悬空链接。

------

## 13. 为什么硬链接不能跨文件系统？

因为 inode 号只在单个文件系统内部唯一。

不同文件系统中的 inode 号没有统一意义。

------

## 14. mount 的作用是什么？

把一个文件系统挂载到已有目录树中的某个目录上，让多个文件系统统一成一棵目录树。

------

# 十七、一句话总结

UNIX 文件系统的核心思想是：文件是字节数组，目录负责把文件名映射到 inode，inode 保存文件元数据和数据块位置；程序通过 `open()` 获得文件描述符，再用 `read()`、`write()`、`lseek()`、`fsync()` 等接口访问文件；`unlink()` 删除的是名字到 inode 的链接，硬链接让多个名字指向同一个 inode，软链接保存目标路径，而 `mount` 把多个底层文件系统统一到一棵目录树中。
