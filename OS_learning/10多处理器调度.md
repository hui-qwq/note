# 第10章 多处理器调度笔记

## 1. 本章核心问题

本章讨论的是：

> 操作系统如何在多个 CPU 上调度进程？

单处理器调度只需要考虑：

> 下一个让哪个进程运行？

而多处理器调度还要考虑：

> 让哪个进程在哪个 CPU 上运行？
> 进程要不要在 CPU 之间迁移？
> 迁移会不会破坏缓存？
> 多个 CPU 同时访问调度队列会不会产生同步问题？

因此，多处理器调度比单处理器调度复杂得多。

------

## 2. 多处理器架构背景

现代多核处理器通常包含多个 CPU 核，每个 CPU 核都有自己的缓存。

一个简化结构如下：

```text
CPU0        CPU1
 L1          L1
 L2          L2
   \        /
    共享 L3
      |
     内存
```

一般来说：

| 缓存层级 | 特点                   |
| -------- | ---------------------- |
| L1 Cache | 每个核心私有，速度最快 |
| L2 Cache | 通常每个核心私有       |
| L3 Cache | 多个核心共享           |
| 内存     | 容量大，但速度慢       |

缓存的存在是为了利用程序的局部性。

------

## 3. 局部性

缓存之所以有效，是因为程序通常具有局部性。

### 3.1 时间局部性

如果一个数据刚刚被访问过，那么它很可能在不久后再次被访问。

例如：

```c
for (int i = 0; i < 1000; i++) {
    sum += x;
}
```

变量 `x` 会被反复访问。

### 3.2 空间局部性

如果程序访问了某个地址的数据，那么它很可能接着访问附近地址的数据。

例如：

```c
for (int i = 0; i < n; i++) {
    sum += a[i];
}
```

数组元素连续存储，访问 `a[i]` 后很可能访问 `a[i+1]`。

------

## 4. 缓存一致性问题

多 CPU 系统中，每个 CPU 可能缓存了同一份内存数据。

例如：

```text
内存中 A = 10
CPU0 缓存中 A = 10
CPU1 缓存中 A = 10
```

如果 CPU0 修改了 A：

```text
CPU0 缓存中 A = 20
内存中 A 可能还是 10
CPU1 缓存中 A 可能还是 10
```

这时如果 CPU1 继续使用旧值，就会出错。

这个问题叫：

> 缓存一致性问题 cache coherence

硬件会通过缓存一致性协议解决这个问题，保证多个 CPU 最终看到一致的数据。

------

## 5. 总线窥探

一种常见的缓存一致性方法是：

> 总线窥探 bus snooping

它的思想是：

> 每个 CPU 的缓存控制器都会监听总线上的内存访问请求。

当 CPU0 修改某个地址时，它会在总线上发出消息：

```text
我要修改地址 A，其他 CPU 缓存中的 A 作废。
```

其他 CPU 发现自己缓存里也有 A，就会把对应缓存块标记为无效。

例如：

```text
CPU0 缓存：A = 20
CPU1 缓存：A = invalid
```

之后 CPU1 再访问 A 时，就不能使用旧缓存，必须重新获取最新值。

------

## 6. 缓存一致性不等于线程安全

硬件缓存一致性只能保证：

> 每个 CPU 看到的数据值是合理的。

但它不能保证：

> 多个线程同时修改共享数据结构时逻辑正确。

例如：

```c
count++;
```

这条语句看起来是一条操作，但实际上可能分成三步：

```text
1. 读取 count
2. count + 1
3. 写回 count
```

如果两个 CPU 同时执行 `count++`：

| 时间 | CPU0         | CPU1         |
| ---- | ------------ | ------------ |
| 1    | 读 count = 0 |              |
| 2    |              | 读 count = 0 |
| 3    | 算出 1       |              |
| 4    |              | 算出 1       |
| 5    | 写回 1       |              |
| 6    |              | 写回 1       |

最后 `count` 可能是 1，而不是 2。

这就是：

> 竞态条件 race condition

------

## 7. 同步

同步的目的就是：

> 多个 CPU 或线程访问共享数据时，保证关键操作不会互相打架。

常见同步机制包括：

- 锁
- 信号量
- 条件变量
- 原子操作

例如用互斥锁保护共享变量：

```c
pthread_mutex_lock(&lock);

count++;

pthread_mutex_unlock(&lock);
```

被锁保护的代码区域叫：

> 临界区 critical section

核心思想是：

> 同一时刻只允许一个线程进入临界区。

------

## 8. 链表删除中的同步问题

教材中的链表删除代码：

```c
Node_t *tmp = head;
int value = head->value;
head = head->next;
free(tmp);
return value;
```

如果两个线程同时执行这段代码，可能会同时拿到同一个 `head`，然后重复删除同一个节点。

正确做法是加锁：

```c
pthread_mutex_lock(&m);

Node_t *tmp = head;
int value = head->value;
head = head->next;
free(tmp);

pthread_mutex_unlock(&m);
return value;
```

这样可以保证一次只有一个线程修改链表。

------

## 9. 缓存亲和度

缓存亲和度，英文是：

> cache affinity

意思是：

> 一个进程如果一直在同一个 CPU 上运行，会更容易利用该 CPU 缓存中的数据，从而运行更快。

例如进程 A 一直在 CPU0 上运行：

```text
CPU0 缓存中可能保存了：
A 的代码
A 的栈数据
A 的数组数据
A 的页表相关信息
```

如果下次 A 继续在 CPU0 上运行，它可以直接使用这些缓存数据。

但如果 A 被迁移到 CPU1：

```text
CPU1 缓存中没有 A 的数据
```

那么 CPU1 需要重新从内存加载数据，性能会下降。

所以调度器通常希望：

> 尽量让进程继续在原来的 CPU 上运行。

------

## 10. 单队列多处理器调度 SQMS

SQMS 全称：

> Single Queue Multiprocessor Scheduling
> 单队列多处理器调度

它的结构是：

```text
        全局就绪队列
      A B C D E F G
       /    |    \
    CPU0  CPU1  CPU2
```

所有 CPU 从同一个全局队列中取任务运行。

------

## 11. SQMS 的优点

SQMS 最大优点是：

> 简单。

它可以直接复用单处理器调度策略。

例如：

- FIFO
- RR
- MLFQ
- 优先级调度

原来单 CPU 是一个队列服务一个 CPU，现在只是让一个队列服务多个 CPU。

------

## 12. SQMS 的缺点一：锁竞争严重

因为多个 CPU 都要访问同一个全局队列，所以必须加锁。

```c
lock(queue);

从队列中取任务;

unlock(queue);
```

否则多个 CPU 可能同时取到同一个任务，造成错误。

但是 CPU 数量越多，锁竞争越严重：

```text
CPU0 等锁
CPU1 等锁
CPU2 等锁
CPU3 等锁
```

这会导致系统花大量时间在同步开销上，而不是执行真正的任务。

因此，SQMS 的可扩展性较差。

------

## 13. SQMS 的缺点二：缓存亲和度差

由于所有 CPU 都从全局队列中取任务，一个进程可能频繁在不同 CPU 间迁移。

例如有 5 个任务 A、B、C、D、E 和 4 个 CPU。

第一轮：

```text
CPU0: A
CPU1: B
CPU2: C
CPU3: D
```

第二轮：

```text
CPU0: E
CPU1: A
CPU2: B
CPU3: C
```

第三轮：

```text
CPU0: D
CPU1: E
CPU2: A
CPU3: B
```

可以看到，进程不断换 CPU 运行，导致原来缓存中的数据无法被很好利用。

这会破坏缓存亲和度。

------

## 14. SQMS 小结

SQMS 的特点可以总结为：

| 方面       | 表现                         |
| ---------- | ---------------------------- |
| 实现复杂度 | 简单                         |
| 负载均衡   | 较好                         |
| 可扩展性   | 较差                         |
| 缓存亲和度 | 较差                         |
| 主要问题   | 全局队列锁竞争，任务频繁迁移 |

一句话：

> SQMS 简单，但随着 CPU 数量增加，锁竞争和缓存亲和度问题会越来越严重。

------

## 15. 多队列多处理器调度 MQMS

MQMS 全称：

> Multi-Queue Multiprocessor Scheduling
> 多队列多处理器调度

它的基本思想是：

> 每个 CPU 拥有自己的调度队列。

例如：

```text
CPU0 队列：A C
CPU1 队列：B D
```

CPU0 只从自己的队列中取任务，CPU1 也只从自己的队列中取任务。

------

## 16. MQMS 的优点一：可扩展性好

由于每个 CPU 有自己的队列，大多数情况下 CPU 不需要争抢同一个全局队列锁。

SQMS：

```text
所有 CPU 抢一个队列
```

MQMS：

```text
CPU0 管自己的队列
CPU1 管自己的队列
CPU2 管自己的队列
```

因此，MQMS 的锁竞争更少，可扩展性更好。

------

## 17. MQMS 的优点二：缓存亲和度好

在 MQMS 中，一个任务通常会固定在某个 CPU 的队列上。

例如：

```text
CPU0 队列：A C
CPU1 队列：B D
```

A 通常一直在 CPU0 上运行，B 通常一直在 CPU1 上运行。

这样任务可以持续利用原 CPU 缓存中的数据。

因此，MQMS 具有较好的缓存亲和度。

------

## 18. MQMS 的缺点：负载不均

MQMS 的主要问题是：

> 负载不均 load imbalance

例如一开始：

```text
CPU0 队列：A C
CPU1 队列：B D
```

这时比较均衡。

如果 C 运行结束：

```text
CPU0 队列：A
CPU1 队列：B D
```

如果每个 CPU 都使用轮转调度，那么：

```text
CPU0：A A A A A A
CPU1：B D B D B D
```

这时 A 独占 CPU0，而 B 和 D 共享 CPU1。

结果是：

> A 得到的 CPU 时间比 B 和 D 多。

这就不公平。

更严重的情况是：

```text
CPU0 队列：空
CPU1 队列：B D
```

这时 CPU0 空闲，而 CPU1 还很忙，系统资源没有被充分利用。

------

## 19. 迁移

解决 MQMS 负载不均的一种方法是：

> 迁移 migration

也就是把任务从繁忙 CPU 的队列迁移到空闲或较轻负载 CPU 的队列。

例如：

```text
CPU0 队列：空
CPU1 队列：B D
```

可以把 B 迁移到 CPU0：

```text
CPU0 队列：B
CPU1 队列：D
```

这样两个 CPU 都有任务可以运行。

------

## 20. 迁移的代价

迁移可以改善负载均衡，但也有代价。

如果任务 B 原来一直在 CPU1 上运行：

```text
CPU1 缓存中有 B 的数据
```

现在 B 被迁移到 CPU0：

```text
CPU0 缓存中没有 B 的数据
```

那么 CPU0 需要重新加载 B 的相关数据，导致缓存命中率下降。

因此，多处理器调度的核心矛盾之一是：

> 负载均衡 vs 缓存亲和度

为了负载均衡，需要迁移任务；
但迁移太频繁，又会破坏缓存亲和度。

------

## 21. 工作窃取

工作窃取，英文是：

> work stealing

它的思想是：

> 任务少的 CPU 主动去查看其他 CPU 的队列，如果发现别的队列任务很多，就从那里偷一个任务过来。

例如：

```text
CPU0 队列：空
CPU1 队列：B D E
```

CPU0 发现自己没任务，就去 CPU1 队列偷一个任务：

```text
CPU0 队列：B
CPU1 队列：D E
```

这样系统负载更加均衡。

------

## 22. 工作窃取的问题

工作窃取也有开销。

如果检查太频繁：

```text
CPU0 经常查看 CPU1
CPU1 经常查看 CPU2
CPU2 经常查看 CPU3
```

那么检查本身会带来同步开销和缓存开销，降低系统可扩展性。

如果检查太少：

```text
很久才检查一次
```

负载不均可能持续很久。

因此工作窃取需要选择合适的检查频率和迁移阈值。

这也是调度器设计中比较困难的地方。

------

## 23. SQMS 和 MQMS 对比

| 调度方式 | 核心思想              | 优点                     | 缺点                     |
| -------- | --------------------- | ------------------------ | ------------------------ |
| SQMS     | 所有 CPU 共用一个队列 | 简单，负载均衡较容易     | 锁竞争严重，缓存亲和度差 |
| MQMS     | 每个 CPU 一个队列     | 可扩展性好，缓存亲和度好 | 容易负载不均，需要迁移   |

简化理解：

```text
SQMS：
一个大队列，所有 CPU 抢任务。
简单，但容易抢锁，任务乱跑。

MQMS：
每个 CPU 一个小队列。
缓存友好，扩展性好，但容易有的 CPU 忙、有的 CPU 闲。
```

------

## 24. Linux 多处理器调度

Linux 历史上出现过多种调度器，例如：

| 调度器      | 特点                          |
| ----------- | ----------------------------- |
| O(1) 调度器 | 多队列，基于优先级，类似 MLFQ |
| CFS         | 多队列，比例公平思想          |
| BFS         | 单队列，比例调度思想          |

这说明：

> 单队列和多队列都不是绝对正确或错误，真实操作系统会根据目标进行复杂权衡。

Linux 调度器设计不仅要考虑公平性，还要考虑：

- 响应时间
- 吞吐量
- 多核扩展性
- 缓存亲和度
- 负载均衡
- 实时任务
- 交互任务

------

## 25. 本章总结

本章介绍了多处理器调度的基本问题。

单处理器调度主要考虑：

```text
公平性
响应时间
周转时间
吞吐量
```

多处理器调度还需要额外考虑：

```text
缓存一致性
同步开销
锁竞争
缓存亲和度
负载均衡
任务迁移
可扩展性
```

核心结论：

> 多处理器调度没有简单答案，本质上是在多个目标之间做权衡。

最重要的权衡是：

```text
负载均衡 vs 缓存亲和度
```

任务不迁移，缓存亲和度好，但可能负载不均。
任务频繁迁移，负载可能更均衡，但缓存亲和度会变差。

------

## 26. 考试重点

### 题目1：多处理器调度相比单处理器调度有什么新问题？

答：

多处理器调度除了要考虑传统的公平性、响应时间、周转时间等指标外，还要考虑缓存一致性、同步开销、缓存亲和度、负载均衡和任务迁移等问题。由于每个 CPU 通常有自己的缓存，如果进程频繁在 CPU 之间迁移，会破坏缓存局部性，降低性能。同时，多个 CPU 访问共享调度队列或共享数据结构时需要同步，会产生锁竞争和额外开销。

------

### 题目2：什么是缓存亲和度？

答：

缓存亲和度是指进程倾向于继续在之前运行过的 CPU 上运行。因为该进程之前使用过的数据可能仍然保存在该 CPU 的缓存中，如果继续在同一个 CPU 上运行，可以提高缓存命中率，减少内存访问开销。如果频繁迁移到其他 CPU，则需要重新加载数据，降低性能。

------

### 题目3：SQMS 有什么优缺点？

答：

SQMS 使用一个全局就绪队列，所有 CPU 都从该队列中取任务运行。它的优点是实现简单，可以直接复用单处理器调度策略，并且负载均衡相对容易。缺点是多个 CPU 访问同一个队列时需要加锁，会产生严重的锁竞争，导致可扩展性差。同时，任务可能频繁在不同 CPU 之间迁移，破坏缓存亲和度。

------

### 题目4：MQMS 有什么优缺点？

答：

MQMS 为每个 CPU 维护一个独立的调度队列。它的优点是减少了对全局队列的竞争，因此可扩展性好；同时任务通常固定在某个 CPU 上运行，可以更好地利用缓存亲和度。缺点是容易产生负载不均，例如某些 CPU 队列任务很多，而另一些 CPU 队列为空。为了解决负载不均，需要使用任务迁移或工作窃取。

------

### 题目5：什么是工作窃取？

答：

工作窃取是一种解决多队列调度中负载不均的方法。任务较少或空闲的 CPU 会检查其他 CPU 的调度队列，如果发现某个队列任务明显较多，就从该队列中迁移一个或多个任务到自己的队列中，从而实现负载均衡。

------

### 题目6：多处理器调度的核心矛盾是什么？

答：

多处理器调度的核心矛盾是负载均衡和缓存亲和度之间的权衡。为了让所有 CPU 都充分工作，调度器需要将任务从繁忙 CPU 迁移到空闲 CPU；但频繁迁移会破坏任务在原 CPU 上形成的缓存状态，降低缓存命中率。因此调度器既不能完全不迁移，也不能过度迁移。

------

## 27. 一句话总结

> 多处理器调度的核心难点不是简单地把任务平均分给多个 CPU，而是在负载均衡、缓存亲和度、同步开销和可扩展性之间做权衡。
