# 第36章 I/O设备：设备交互与驱动程序

## 一、本章核心问题

本章主要讨论操作系统如何与 I/O 设备交互。

核心问题有四个：

| 问题                         | 解决方式                |
| ---------------------------- | ----------------------- |
| CPU 如何知道设备完成了？     | 轮询 / 中断             |
| CPU 如何减少数据搬运开销？   | DMA                     |
| 操作系统如何访问设备寄存器？ | I/O 指令 / 内存映射 I/O |
| 如何屏蔽不同设备的差异？     | 设备驱动程序            |

一句话总结：

> 中断解决“CPU 别傻等”，DMA 解决“CPU 别亲自搬”，I/O 指令和 MMIO 解决“怎么和设备说话”，设备驱动解决“怎么统一管理不同设备”。

------

## 二、设备交互的方法

操作系统要控制设备，本质上是读写设备暴露出来的寄存器。

常见的设备寄存器有：

| 寄存器             | 作用             |
| ------------------ | ---------------- |
| status 状态寄存器  | 查看设备当前状态 |
| command 命令寄存器 | 向设备发送命令   |
| data 数据寄存器    | 传输数据         |

操作系统访问这些寄存器主要有两种方式：

1. I/O 指令
2. 内存映射 I/O

------

## 三、I/O 指令

第一种方式是使用专门的 I/O 指令。

在 x86 中，常见的 I/O 指令有：

```asm
in
out
```

它们用于从设备端口读数据，或者向设备端口写数据。

例如：

```text
out 0x1F7, READ_COMMAND
```

可以理解为：

> 向 0x1F7 这个设备端口写入一个读命令。

这些 I/O 指令通常是特权指令，只有操作系统内核可以执行。

原因是如果普通用户程序也能直接操作设备，就会非常危险。例如，用户程序可能直接读写磁盘，绕过文件系统和权限控制，甚至控制整台机器。

所以设备访问必须由操作系统统一管理。

------

## 四、内存映射 I/O

第二种方式是内存映射 I/O，英文是：

```text
Memory-Mapped I/O，MMIO
```

它的核心思想是：

> 把设备寄存器映射到某些特殊的内存地址上。

这样操作系统访问设备时，就像访问普通内存一样：

```c
*device_register = command;
status = *device_status;
```

但硬件会识别这些特殊地址，并不会真的访问内存，而是把读写操作转发给对应设备。

也就是说：

```text
普通内存访问：
CPU 读写地址 → 内存

内存映射 I/O：
CPU 读写特殊地址 → 设备寄存器
```

------

## 五、I/O 指令与内存映射 I/O 对比

| 方式         | 核心思想                           | 特点                   |
| ------------ | ---------------------------------- | ---------------------- |
| I/O 指令     | 使用专门的 in/out 指令访问设备端口 | 需要专门指令           |
| 内存映射 I/O | 把设备寄存器映射成内存地址         | 像访问内存一样访问设备 |

两种方式今天都在使用。

一句话理解：

> I/O 指令是专门开一套设备访问通道，MMIO 是把设备寄存器伪装成内存地址。

------

## 六、设备驱动程序

不同设备的接口差异很大。

例如文件系统可能运行在不同设备之上：

```text
SCSI 硬盘
IDE 硬盘
SATA 磁盘
USB U盘
SSD
```

文件系统不应该关心每种设备具体要写哪个寄存器、发送什么命令、如何等待完成。

文件系统只希望看到统一的接口：

```text
读某个块
写某个块
```

为了解决这个问题，操作系统引入了设备驱动程序。

设备驱动程序的作用是：

> 封装具体设备的底层细节，对上层提供统一接口。

------

## 七、Linux 文件系统栈

Linux 中大致可以这样理解：

```text
应用程序
  ↓
文件系统
  ↓
通用块设备层
  ↓
设备驱动程序
  ↓
具体硬件设备
```

文件系统不直接控制磁盘，而是向通用块设备层发送读写请求。

通用块设备层再把请求交给具体设备驱动程序。

最后由设备驱动程序真正操作硬件。

这样做的好处是：

> 文件系统可以做到设备无关。

也就是说，文件系统不需要知道底层是 IDE 磁盘、SCSI 磁盘，还是 USB 存储设备。

------

## 八、抽象的代价

设备驱动程序和通用接口带来了抽象，但抽象也有代价。

有些设备可能有特殊能力，但为了适配统一接口，这些特殊能力可能无法完全暴露给上层。

例如 SCSI 设备可以提供非常丰富的错误信息。

但如果上层通用块设备接口只支持简单错误码，例如：

```text
EIO：一般 I/O 错误
```

那么 SCSI 的详细错误信息就可能被隐藏掉。

这就是抽象的常见折中：

```text
统一性增强了
但设备的特殊能力可能被屏蔽了
```

------

## 九、为什么驱动程序代码很多？

在 Linux 和 Windows 这样的操作系统中，驱动程序代码占比非常高。

原因很简单：

> 设备种类太多了。

不同厂商、不同型号、不同接口的设备都需要对应驱动。

驱动程序也容易成为系统 bug 的来源，因为：

1. 硬件细节复杂
2. 设备种类繁多
3. 驱动开发者水平参差不齐
4. 驱动运行在内核态
5. 驱动出错可能导致整个系统崩溃

所以很多系统崩溃，并不是调度器、文件系统、内存管理本身的问题，而可能是驱动程序出了问题。

## 十、案例：IDE 磁盘驱动程序

IDE 磁盘通过一组 I/O 端口和操作系统交互。

常见端口如下：

| 地址  | 作用                    |
| ----- | ----------------------- |
| 0x1F0 | 数据端口                |
| 0x1F1 | 错误寄存器              |
| 0x1F2 | 扇区数量                |
| 0x1F3 | LBA 低字节              |
| 0x1F4 | LBA 中字节              |
| 0x1F5 | LBA 高字节              |
| 0x1F6 | 设备编号和 LBA 高位     |
| 0x1F7 | 命令寄存器 / 状态寄存器 |
| 0x3F6 | 控制寄存器              |

其中最重要的是：

```text
0x1F7
```

读 0x1F7 时，它表示状态寄存器。

写 0x1F7 时，它表示命令寄存器。

------

## 十一、IDE 状态寄存器

IDE 状态寄存器中有几个重要标志位：

| 标志位 | 含义         |
| ------ | ------------ |
| BUSY   | 设备忙       |
| READY  | 设备就绪     |
| DRQ    | 设备请求数据 |
| ERROR  | 操作出错     |

驱动程序通常要等待设备满足：

```text
不是 BUSY
并且 READY
```

也就是设备不忙，并且已经准备好接收命令。

------

## 十二、IDE 磁盘读写协议

IDE 磁盘的读写大致流程如下：

```text
1. 等待磁盘就绪
2. 写入请求参数
3. 发送读/写命令
4. 如果是写请求，向数据端口写入数据
5. 等待磁盘完成并触发中断
6. 如果是读请求，从数据端口读取数据
7. 检查是否出错
8. 唤醒等待 I/O 的进程
```

可以画成流程：

```text
等待 READY 且非 BUSY
        ↓
写入扇区数量
        ↓
写入 LBA 地址
        ↓
写入设备编号
        ↓
发送 READ 或 WRITE 命令
        ↓
磁盘执行 I/O
        ↓
磁盘完成后触发中断
        ↓
中断处理程序完成后续工作
```

------

## 十三、xv6 的 IDE 驱动程序

xv6 中的 IDE 驱动程序主要由四个函数组成：

```text
ide_wait_ready()
ide_start_request()
ide_rw()
ide_intr()
```

------

## 十四、ide_wait_ready()

作用：

> 等待磁盘进入就绪状态。

逻辑类似：

```c
while (设备忙 || 设备没准备好)
    ;
```

它本质上是在轮询设备状态寄存器。

目的：

> 在发送命令之前，先确认设备能够接收命令。

------

## 十五、ide_start_request()

作用：

> 真正向磁盘发起一个读写请求。

它主要完成以下工作：

```text
1. 等待设备就绪
2. 写控制寄存器
3. 写扇区数量
4. 写 LBA 地址
5. 写设备编号
6. 判断是读请求还是写请求
7. 发送 READ 或 WRITE 命令
8. 如果是写请求，立即把数据写入数据端口
```

写请求时，CPU 会把数据写到数据端口：

```text
0x1F0
```

读请求时，驱动先发送读命令，等磁盘完成后由中断处理程序读取数据。

------

## 十六、ide_rw()

作用：

> 对外提供磁盘读写接口，并把请求加入请求队列。

它的核心逻辑是：

```text
1. 获取锁
2. 将请求加入 IDE 请求队列
3. 如果队列原来为空，立即启动该请求
4. 当前进程 sleep，等待 I/O 完成
5. 请求完成后被 wakeup 唤醒
6. 释放锁
```

为什么需要队列？

因为磁盘不能同时处理任意多个请求。

多个进程同时请求磁盘时，需要排队：

```text
请求1 → 请求2 → 请求3 → ...
```

------

## 十七、ide_intr()

作用：

> 当磁盘完成 I/O 后触发中断，由 ide_intr() 处理中断。

它主要完成以下工作：

```text
1. 获取锁
2. 找到当前完成的请求
3. 如果是读请求，从数据端口读取数据
4. 标记缓冲区有效
5. 清除脏标记
6. 唤醒等待该请求的进程
7. 如果队列中还有请求，启动下一个请求
8. 释放锁
```

它的核心思想是：

```text
当前请求完成 → 唤醒等待进程 → 启动下一个请求
```

------

## 十八、IDE 驱动整体流程

完整过程可以总结为：

```text
进程调用 ide_rw()
        ↓
请求加入 ide_queue
        ↓
如果磁盘空闲，调用 ide_start_request()
        ↓
向 IDE 磁盘写入参数和命令
        ↓
进程 sleep()
        ↓
磁盘执行 I/O
        ↓
磁盘完成后触发中断
        ↓
CPU 执行 ide_intr()
        ↓
如果是读请求，从数据端口读数据
        ↓
标记请求完成
        ↓
wakeup() 唤醒等待进程
        ↓
如果队列还有请求，启动下一个请求
```

这个过程把前面学过的很多概念串起来了：

```text
锁
队列
睡眠
唤醒
中断
设备寄存器
驱动程序
```

------

## 十九、历史背景

中断和 DMA 都是很早就出现的技术。

它们出现的原因很自然：

```text
CPU 很快
I/O 设备很慢
```

如果 CPU 一直等待 I/O 完成，会浪费大量计算能力。

所以人们很早就想到：

```text
设备慢慢工作
CPU 先去做别的事情
设备完成后再通知 CPU
```

这就是中断和 DMA 背后的基本思想。

## 二十、本章总结

第36章介绍了操作系统与 I/O 设备交互的基本机制。

可以总结成四组核心概念：

### 1. CPU 如何等待设备？

```text
轮询 polling：
CPU 不断检查设备状态

中断 interrupt：
设备完成后主动通知 CPU
```

### 2. 谁负责搬数据？

```text
PIO：
CPU 亲自搬数据

DMA：
DMA 控制器帮 CPU 搬数据
```

### 3. 操作系统如何访问设备寄存器？

```text
I/O 指令：
使用 in/out 等专门指令访问设备端口

内存映射 I/O：
把设备寄存器映射到内存地址
```

### 4. 操作系统如何屏蔽设备差异？

```text
设备驱动程序：
封装具体硬件细节，对上层提供统一接口
```

## 二十一、最终记忆版

```text
轮询：CPU 主动问设备好了没
中断：设备好了主动通知 CPU
PIO：CPU 亲自搬数据
DMA：DMA 控制器帮 CPU 搬数据
I/O 指令：用专门指令访问设备
MMIO：把设备寄存器映射成内存地址
设备驱动：封装硬件差异，提供统一接口
```

一句话总结：

> 操作系统通过 I/O 指令或内存映射 I/O 访问设备寄存器，通过中断和 DMA 提高效率，通过设备驱动程序屏蔽不同硬件设备的差异。
