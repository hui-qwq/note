# 第 4 章：抽象——进程

## 1. 本章核心

本章主要讲操作系统中最基本的抽象之一：**进程**。

进程可以简单理解为：

> 进程 = 正在运行的程序

程序本身只是存放在磁盘上的代码和数据，它没有生命周期。只有当操作系统把它加载到内存中，并为它创建运行环境后，它才变成一个真正运行中的进程。

------

## 2. 为什么需要进程？

现实中，我们希望计算机可以同时运行很多程序：

- 浏览器
- 音乐播放器
- 编辑器
- 游戏
- 后台服务
- 终端程序

但物理 CPU 数量是有限的。操作系统通过 **CPU 虚拟化**，让每个程序看起来都像拥有自己的 CPU。

核心方法是：

> 时分共享 CPU

也就是让每个进程运行一小段时间，然后切换到另一个进程。因为切换速度很快，所以用户感觉多个程序好像在同时运行。

------

## 3. 机制与策略

操作系统中经常区分两个概念：

### 机制 mechanism

机制解决的是：

> 怎么做？

例如：

- 如何保存一个进程的现场？
- 如何恢复一个进程？
- 如何进行上下文切换？

典型机制：

> 上下文切换 context switch

------

### 策略 policy

策略解决的是：

> 选哪个？

例如：

- 当前应该运行哪个进程？
- 哪个进程优先级更高？
- I/O 完成后要不要马上切回原进程？

典型策略：

> 调度策略 scheduling policy

一句话总结：

> 机制负责“怎么切换”，策略负责“切给谁”。

------

## 4. 进程的机器状态

进程不是只有代码，它还包含运行时的状态。

一个进程的机器状态主要包括：

### 1. 地址空间

也就是进程能访问的内存，包括：

- 代码段
- 静态数据
- 堆
- 栈
- 动态库等

### 2. 寄存器

进程运行时，CPU 寄存器中保存了很多重要信息，例如：

- PC / IP：当前执行到哪条指令
- SP：栈指针
- FP / BP：栈帧指针
- 通用寄存器

如果进程被暂停，操作系统必须保存这些寄存器。否则下次恢复时，进程就不知道从哪里继续执行。

### 3. I/O 信息

进程还可能打开了一些文件或设备，例如：

- 标准输入
- 标准输出
- 标准错误
- 普通文件
- socket
- 管道

这些也属于进程运行状态的一部分。

------

## 5. 进程 API

操作系统通常会提供一些操作进程的接口：

| 操作    | 含义                   |
| ------- | ---------------------- |
| create  | 创建进程               |
| destroy | 销毁进程               |
| wait    | 等待进程结束           |
| control | 暂停、恢复、杀死等控制 |
| status  | 查看进程状态           |

在 UNIX/Linux 中，常见进程相关函数包括：

```c
fork();      // 创建子进程
execve();    // 加载并运行新程序
exit();      // 退出进程
wait();      // 等待子进程结束
waitpid();   // 等待指定子进程
kill();      // 向进程发送信号
```

------

## 6. 程序如何变成进程？

程序变成进程，大致需要以下步骤：

```text
磁盘上的程序
    ↓
加载代码和静态数据到内存
    ↓
创建地址空间
    ↓
分配栈
    ↓
分配堆
    ↓
初始化 I/O
    ↓
跳转到入口点
    ↓
程序开始运行
```

### 1. 加载代码和静态数据

程序原本在磁盘或 SSD 上，操作系统需要把它的代码和静态数据加载到内存中。

现代操作系统通常采用 **惰性加载**：

> 程序不一定一开始全部加载进内存，而是用到哪部分再加载哪部分。

这和后面要学的虚拟内存、分页、缺页异常有关。

### 2. 初始化栈

栈用于保存：

- 函数参数
- 局部变量
- 返回地址
- 函数调用信息

例如：

```c
int main(int argc, char *argv[])
```

其中 `argc` 和 `argv` 也需要操作系统在进程启动时准备好。

### 3. 初始化堆

堆用于动态内存分配，例如：

```c
int *p = malloc(sizeof(int) * 100);
```

刚开始堆一般较小，程序运行过程中可以通过 `malloc()` 继续申请更多空间。

### 4. 初始化 I/O

在 UNIX/Linux 中，每个进程默认有 3 个文件描述符：

| 文件描述符 | 含义            |
| ---------- | --------------- |
| 0          | 标准输入 stdin  |
| 1          | 标准输出 stdout |
| 2          | 标准错误 stderr |

所以 `printf()` 默认输出到屏幕，本质上就是写到文件描述符 1。

------

## 7. 进程状态

进程不会一直运行，它会在不同状态之间切换。

最基本的 3 个状态是：

| 状态    | 含义                           |
| ------- | ------------------------------ |
| Running | 正在 CPU 上运行                |
| Ready   | 已准备好运行，但还没轮到 CPU   |
| Blocked | 正在等待某个事件，暂时不能运行 |

状态转换如下：

```text
Ready → Running
进程被调度，获得 CPU

Running → Ready
时间片用完，或者被操作系统切走

Running → Blocked
进程发起 I/O 或等待某个事件

Blocked → Ready
等待的事件完成，可以继续运行
```

例如，当一个进程发起磁盘 I/O 时，它会进入阻塞状态。此时 CPU 不应该傻等，而应该去运行其他就绪进程，从而提高 CPU 利用率。

------

## 8. xv6 中的进程数据结构

操作系统需要用数据结构记录每个进程的信息。

这种结构通常叫：

> PCB：Process Control Block，进程控制块

在 xv6 中，对应的是：

```c
struct proc
```

它记录了进程的关键信息，例如：

- 进程内存起始位置
- 进程内存大小
- 内核栈
- 进程状态
- PID
- 父进程
- 等待通道
- 是否被杀死
- 打开的文件
- 当前工作目录
- 上下文
- trapframe

简化理解：

```text
PCB 保存了操作系统管理一个进程所需的全部信息。
```

------

## 9. context：上下文

xv6 中有一个结构：

```c
struct context {
  int eip;
  int esp;
  int ebx;
  int ecx;
  int edx;
  int esi;
  int edi;
  int ebp;
};
```

它用于保存进程被暂停时的寄存器状态。

重要寄存器：

| 寄存器 | 含义                             |
| ------ | -------------------------------- |
| eip    | 指令指针，表示当前执行到哪条指令 |
| esp    | 栈指针，表示当前栈顶             |
| ebp    | 栈帧指针，用于管理函数调用栈帧   |

当操作系统进行上下文切换时，会保存当前进程的 context，并恢复另一个进程的 context。

一句话：

> context 保存的是进程暂停时的 CPU 现场。

------

## 10. xv6 中的进程状态

xv6 中进程状态包括：

```c
enum proc_state {
    UNUSED,
    EMBRYO,
    SLEEPING,
    RUNNABLE,
    RUNNING,
    ZOMBIE
};
```

含义如下：

| 状态     | 含义                         |
| -------- | ---------------------------- |
| UNUSED   | 进程槽位未使用               |
| EMBRYO   | 进程正在创建中               |
| SLEEPING | 进程阻塞，正在等待事件       |
| RUNNABLE | 进程就绪，可以运行           |
| RUNNING  | 进程正在运行                 |
| ZOMBIE   | 进程已结束，但父进程还没回收 |

------

## 11. 僵尸进程

僵尸进程是：

> 已经结束，但还没有被父进程回收的进程。

它存在的原因是：父进程可能需要获取子进程的退出状态。

例如：

```c
wait();
waitpid();
```

父进程调用 `wait()` 后，操作系统就可以清理子进程留下的数据结构。

僵尸状态不是错误设计，而是 UNIX 系统中正常的一部分。

但是，如果父进程一直不回收子进程，僵尸进程就会一直残留。

------

## 12. process list：进程列表

操作系统会维护一个进程列表，用来跟踪系统中所有进程。

可以粗略理解为：

```c
struct proc proc_table[NPROC];
```

每一个 `struct proc` 对应一个进程。

操作系统通过进程列表知道：

- 哪些进程正在运行
- 哪些进程处于就绪状态
- 哪些进程正在阻塞
- 哪些进程已经结束
- 哪些进程需要被唤醒
- 哪些进程需要被清理

------

## 13. 本章重点总结

本章主要回答了一个问题：

> 操作系统如何把一个普通程序包装成一个可管理、可调度、可暂停、可恢复的执行实体？

答案包括：

```text
进程抽象
进程状态
进程 API
进程控制块 PCB
上下文切换
调度策略
```

最重要的几句话：

1. 程序是磁盘上的静态文件，进程是正在运行的程序。
2. 操作系统通过时分共享 CPU，制造出多个进程同时运行的假象。
3. 进程有自己的机器状态，包括内存、寄存器和 I/O 信息。
4. 进程会在 Running、Ready、Blocked 等状态之间切换。
5. 操作系统用 PCB 保存每个进程的全部关键信息。
6. 上下文切换的本质是保存和恢复寄存器现场。
7. 僵尸进程是已经结束但尚未被父进程回收的进程。

------

## 14. 作业理解方向

`process-run.py` 模拟器主要用来观察进程状态变化。

需要重点理解：

### CPU 利用率

如果所有进程都只使用 CPU，没有 I/O 阻塞，那么 CPU 利用率通常是 100%。

例如：

```bash
./process-run.py -l 5:100,5:100
```

两个进程都只执行 CPU 指令，所以 CPU 一直忙。

------

### I/O 阻塞

如果一个进程发起 I/O，它会进入阻塞状态。

此时有两种策略：

```text
SWITCH_ON_IO
进程发起 I/O 后，立刻切换到其他进程运行。

SWITCH_ON_END
进程发起 I/O 后，不切换，CPU 等待 I/O 完成。
```

一般来说：

> SWITCH_ON_IO 的 CPU 利用率更高。

因为 CPU 不会傻等 I/O。

------

### I/O 完成后的调度

I/O 完成后，也有两种策略：

```text
IO_RUN_IMMEDIATE
I/O 完成后，马上优先运行刚完成 I/O 的进程。

IO_RUN_LATER
I/O 完成后，不马上运行它，而是让当前进程继续运行。
```

如果系统重视交互响应，`IO_RUN_IMMEDIATE` 往往更好。
如果系统想减少切换开销，也可能选择 `IO_RUN_LATER`。

------

## 15. 一句话总览

> 进程是操作系统对正在运行程序的抽象。操作系统通过 PCB 记录进程状态，通过上下文切换暂停和恢复进程，通过调度策略决定哪个进程运行，从而实现 CPU 虚拟化。

## 三、参考答案

### 1. 什么是进程？

进程是正在运行的程序。

更完整地说：

> 进程 = 程序代码 + 当前运行状态 + 操作系统为它维护的资源信息。

------

### 2. 程序和进程有什么区别？

程序是静态的，进程是动态的。

程序只是磁盘上的文件，例如：

```text
a.out
chrome.exe
```

进程是程序被加载到内存并开始运行后的执行实体。

一句话：

> 程序是“菜谱”，进程是“正在做菜的过程”。

------

### 3. 操作系统为什么要引入“进程”这个抽象？

为了让多个程序可以被统一管理、调度和隔离。

操作系统通过进程抽象，可以管理：

```text
谁在运行
谁在等待
谁占用哪些资源
谁该被调度
谁该被终止
```

------

### 4. CPU 虚拟化是什么意思？

CPU 虚拟化是指：

> 操作系统让每个进程都感觉自己好像独占 CPU。

实际上物理 CPU 数量有限，操作系统通过快速切换多个进程，制造出“多个程序同时运行”的假象。

------

### 5. 什么是时分共享？

时分共享就是：

> 多个进程轮流使用 CPU，每个进程运行一小段时间。

例如：

```text
进程 A 跑 10ms
进程 B 跑 10ms
进程 C 跑 10ms
再切回进程 A
```

因为切换很快，所以用户感觉它们在同时运行。

------

### 6. 进程的机器状态主要包括哪些内容？

主要包括：

```text
内存 / 地址空间
寄存器
程序计数器 PC / IP
栈指针 SP
打开的文件
I/O 状态
```

简单说，就是进程运行到一半时，必须保存下来、以后才能继续运行的信息。

------

### 7. 为什么操作系统在切换进程时要保存寄存器？

因为寄存器保存了进程当前的执行现场。

如果不保存寄存器，进程下次恢复运行时就不知道：

```text
执行到哪条指令
栈在哪里
临时变量是什么
函数调用到哪一步
```

所以切换前必须保存，恢复时再放回 CPU。

------

### 8. 程序计数器 PC / 指令指针 IP 的作用是什么？

PC / IP 用来记录：

> CPU 当前正在执行哪条指令，或者下一条要执行哪条指令。

上下文切换时必须保存 PC / IP，否则进程恢复后不知道从哪里继续执行。

------

### 9. 进程创建时，操作系统通常需要做哪些事情？

通常包括：

```text
加载代码和静态数据到内存
创建进程地址空间
分配栈
分配堆
初始化文件描述符
设置寄存器状态
设置程序入口
把进程加入进程列表
```

最后，操作系统把 CPU 控制权交给新进程，程序开始执行。

------

### 10. 栈和堆在进程中分别有什么作用？

栈主要用于：

```text
函数调用
局部变量
函数参数
返回地址
```

堆主要用于：

```text
动态内存分配
malloc/free
new/delete
```

一句话：

> 栈管函数调用，堆管动态申请的内存。

------

### 11. Linux/UNIX 中，每个进程默认打开的 3 个文件描述符是什么？

```text
0：标准输入 stdin
1：标准输出 stdout
2：标准错误 stderr
```

通常：

```text
stdin  对应键盘输入
stdout 对应屏幕普通输出
stderr 对应屏幕错误输出
```

------

### 12. 进程的三种基本状态是什么？

三种基本状态是：

```text
Running：运行态
Ready：就绪态
Blocked：阻塞态
```

------

### 13. Ready 和 Running 有什么区别？

Ready 表示：

> 进程已经准备好运行，但还没有拿到 CPU。

Running 表示：

> 进程正在 CPU 上执行指令。

所以区别是：

```text
Ready：能跑，但没轮到
Running：正在跑
```

------

### 14. 什么情况下进程会进入阻塞状态？

当进程需要等待某个事件时，会进入阻塞状态。

常见情况：

```text
等待磁盘 I/O
等待网络数据
等待用户输入
等待锁
等待子进程结束
等待管道数据
```

阻塞时，进程暂时不能继续运行。

------

### 15. I/O 完成后，阻塞进程会直接变成 Running 吗？为什么？

不一定。

通常是：

```text
Blocked → Ready
```

I/O 完成后，进程只是重新具备运行条件，变成就绪态。

至于它能不能马上变成 Running，要看操作系统调度器是否选择它。

------

### 16. 什么是上下文切换？

上下文切换是指：

> 操作系统暂停当前进程，保存它的运行现场，然后恢复另一个进程的运行现场，让另一个进程继续执行。

核心动作是：

```text
保存当前进程寄存器
更新当前进程状态
选择下一个进程
恢复下一个进程寄存器
跳转执行
```

------

### 17. 什么是 PCB？它的作用是什么？

PCB 是 Process Control Block，中文叫：

> 进程控制块。

它是操作系统用来记录和管理进程的数据结构。

一句话：

> PCB 是进程在操作系统里的“档案”。

------

### 18. PCB 中一般会保存哪些信息？

PCB 通常保存：

```text
PID
进程状态
寄存器上下文
程序计数器
内存信息
打开的文件
父进程信息
优先级
调度信息
退出状态
```

不同操作系统的 PCB 具体字段不同，但核心作用都是记录进程信息。

------

### 19. 什么是僵尸进程？

僵尸进程是：

> 子进程已经结束，但父进程还没有调用 wait/waitpid 回收它。

它已经不运行了，但还保留少量信息，例如：

```text
PID
退出码
终止原因
```

------

### 20. 为什么僵尸进程不能一结束就被操作系统完全清理？

因为父进程可能还需要获取子进程的退出状态。

例如父进程需要知道：

```text
子进程是否正常结束
返回值是多少
是否被信号杀死
```

所以操作系统会暂时保留子进程的一些信息。

等父进程调用 `wait()` 或 `waitpid()` 后，操作系统才会彻底清理它。

------

## 四、核心总结

进程是操作系统对正在运行程序的抽象。

操作系统通过进程来管理程序的运行状态、资源占用和调度顺序。为了让多个程序看起来能够同时运行，操作系统使用时分共享技术，让 CPU 在多个进程之间快速切换。

每个进程都有自己的机器状态，包括内存、寄存器、程序计数器、栈指针、打开的文件等。操作系统用 PCB 保存这些信息，从而实现进程的暂停、恢复、调度和清理。

本章最重要的三个关键词是：

```text
进程状态
PCB
上下文切换
```

一句话总结：

> 进程是正在运行的程序，PCB 是操作系统记录进程信息的数据结构，上下文切换是操作系统在多个进程之间切换 CPU 的核心机制。
