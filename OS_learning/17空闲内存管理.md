# 第17章 空闲空间管理

## 1. 本章核心问题

本章讨论的是内存管理中的一个基础问题：

> 如何管理一块空闲内存空间，以满足不断变化的内存分配和释放请求？

这个问题不仅出现在操作系统中，也出现在用户级内存分配库中，例如：

```c
malloc();
free();
```

无论是 `malloc` 管理进程的堆空间，还是操作系统管理进程地址空间，本质上都需要解决：

```text
哪些空间是空闲的？
如何找到合适的空闲空间？
如何减少碎片？
如何提高分配和释放速度？
```

------

## 2. 固定大小 vs 可变大小

### 2.1 固定大小空间

如果内存被划分为固定大小的单元，例如页：

```text
[4KB][4KB][4KB][4KB]
```

管理会比较简单。

只需要维护一个空闲页列表：

```text
free_list -> page1 -> page3 -> page5
```

申请空间时，直接取出一个空闲页即可。

------

### 2.2 可变大小空间

如果空闲块大小不固定，管理就复杂很多。

例如：

```text
[10B free][10B used][10B free]
```

虽然总空闲空间是：

```text
10B + 10B = 20B
```

但如果请求：

```c
malloc(15);
```

会失败。

因为没有一块连续的 15B 空间。

这就是外部碎片问题。

------

## 3. 外部碎片与内部碎片

### 3.1 外部碎片

外部碎片指的是：

> 空闲空间总量足够，但被分散成很多小块，导致无法满足较大的连续空间请求。

例如：

```text
[10B free][used][10B free]
```

总共有 20B 空闲空间，但无法满足 15B 的连续空间请求。

------

### 3.2 内部碎片

内部碎片指的是：

> 分配器实际分配的空间大于用户请求的空间，多出来但没有被使用的部分就是内部碎片。

例如：

```c
malloc(13);
```

为了对齐，分配器可能实际分配 16B。

```text
申请 13B
实际分配 16B
浪费 3B
```

这 3B 就是内部碎片。

------

## 4. malloc/free 的基本假设

本章主要以用户级内存分配库为例。

接口如下：

```c
void *malloc(size_t size);
void free(void *ptr);
```

### 4.1 malloc

`malloc(size)` 用于申请指定大小的内存空间。

例如：

```c
void *p = malloc(100);
```

表示申请至少 100 字节的空间。

------

### 4.2 free

`free(ptr)` 用于释放之前申请的内存空间。

例如：

```c
free(p);
```

注意，`free()` 只传入指针，没有传入大小。

所以分配器必须自己知道：

```text
这个指针对应的内存块到底有多大？
```

这需要通过额外的元数据实现。

------

## 5. 空闲列表 free list

分配器管理的内存区域通常叫做堆。

为了记录堆中哪些空间是空闲的，分配器会维护一种数据结构：

> 空闲列表 free list

空闲列表中保存所有空闲块的信息，例如：

```text
free_list -> [addr=0, size=10] -> [addr=20, size=10]
```

虽然叫“列表”，但它不一定真的是链表，也可以是树、数组、分离链表等数据结构。

------

# 6. 底层机制

空闲空间管理依赖几个基础机制：

```text
分割 splitting
合并 coalescing
头块 header
嵌入空闲列表 embedded free list
堆增长 heap growth
```

------

## 6.1 分割 splitting

当一个空闲块比用户请求的空间更大时，分配器会将它分成两部分：

```text
一部分分配给用户
另一部分继续留在空闲列表中
```

例如：

```text
原来：
[10B free]

malloc(1) 后：
[1B allocated][9B free]
```

这就是分割。

------

## 6.2 合并 coalescing

分割会让空闲空间越来越碎，所以释放内存时需要合并相邻空闲块。

例如：

```text
[10B free][10B free][10B free]
```

如果不合并，看起来是三个小空闲块。

合并后：

```text
[30B free]
```

这样就可以满足更大的连续空间请求。

------

## 6.3 头块 header

由于 `free(ptr)` 不传入大小，所以分配器需要在用户内存块前面保存一些额外信息。

这个额外信息叫做头块：

```text
[header][用户可用空间]
        ^
        malloc 返回这里
```

一个简单的头块可以这样定义：

```c
typedef struct {
    int size;
    int magic;
} header_t;
```

其中：

| 字段    | 作用                         |
| ------- | ---------------------------- |
| `size`  | 记录用户申请的空间大小       |
| `magic` | 用于完整性检查，防止错误释放 |

------

## 6.4 free 如何找到 header？

假设用户调用：

```c
free(ptr);
```

分配器会通过指针运算找到头块：

```c
header_t *hptr = (void *)ptr - sizeof(header_t);
```

然后通过：

```c
hptr->size
```

知道这块内存的大小。

------

## 6.5 实际分配大小

用户申请的大小不等于分配器实际消耗的大小。

例如用户申请：

```c
malloc(100);
```

如果头块大小是 8B，那么实际需要：

```text
100B 用户空间 + 8B 头块 = 108B
```

所以分配器要寻找至少 108B 的空闲块。

------

## 6.6 嵌入空闲列表

在普通程序中，创建链表节点可能会调用：

```c
malloc(sizeof(node_t));
```

但内存分配器本身就是在实现 `malloc()`，不能依赖自己。

所以空闲列表的节点需要直接放在空闲块内部。

例如：

```c
typedef struct node_t {
    int size;
    struct node_t *next;
} node_t;
```

空闲块本身就可以充当链表节点：

```text
[size][next][剩余空闲空间]
```

------

## 6.7 堆增长

如果堆中没有足够大的空闲块，可以有两种处理方式：

### 方式一：返回失败

```c
return NULL;
```

表示内存申请失败。

### 方式二：向操作系统申请更多空间

传统分配器通常会使用类似：

```c
sbrk()
```

的系统调用让堆增长。

操作系统会将更多物理页映射到进程地址空间中，使堆变大。

流程如下：

```text
malloc 请求空间
    ↓
空闲列表中找不到足够大的块
    ↓
调用 sbrk() 向操作系统申请更多空间
    ↓
操作系统映射更多页
    ↓
堆变大
    ↓
malloc 继续完成分配
```

------

# 7. 基本分配策略

有了空闲列表之后，问题变成：

> 当空闲列表中有多个空闲块时，应该选择哪一块？

常见策略有：

```text
最优匹配 best fit
最差匹配 worst fit
首次匹配 first fit
下次匹配 next fit
```

------

## 7.1 最优匹配 best fit

最优匹配的思想是：

> 遍历整个空闲列表，找到能满足请求的最小空闲块。

例如：

```text
空闲列表：[10B] -> [30B] -> [20B]
请求：malloc(15)
```

能满足请求的块有：

```text
30B, 20B
```

其中 20B 最小，所以选择 20B。

### 优点

空间浪费较少。

### 缺点

需要遍历整个空闲列表，速度较慢。

而且可能产生很多小碎片。

------

## 7.2 最差匹配 worst fit

最差匹配的思想是：

> 每次选择最大的空闲块。

例如：

```text
空闲列表：[10B] -> [30B] -> [20B]
请求：malloc(15)
```

最差匹配会选择 30B。

### 优点

希望保留较大的剩余空间，避免产生太小的碎片。

### 缺点

需要遍历整个列表。

实际表现通常较差，可能造成较多碎片。

------

## 7.3 首次匹配 first fit

首次匹配的思想是：

> 从头开始找，遇到第一个足够大的空闲块就使用它。

例如：

```text
空闲列表：[10B] -> [30B] -> [20B]
请求：malloc(15)
```

10B 不够，30B 够，所以选择 30B。

### 优点

速度较快，不一定需要遍历整个列表。

### 缺点

容易让空闲列表前部产生很多小碎片。

------

## 7.4 下次匹配 next fit

下次匹配是首次匹配的变体。

它不会每次都从链表头部开始查找，而是记录上一次查找结束的位置。

下一次从该位置继续查找。

### 优点

避免反复切割列表开头的空闲块。

### 缺点

效果不一定比首次匹配更好。

------

## 7.5 策略对比

| 策略      | 选择方式             | 是否需要遍历整个列表 | 优点               | 缺点               |
| --------- | -------------------- | -------------------- | ------------------ | ------------------ |
| best fit  | 选择最小但足够大的块 | 通常需要             | 空间浪费较少       | 慢，容易产生小碎片 |
| worst fit | 选择最大的块         | 通常需要             | 希望保留较大剩余块 | 慢，实际表现差     |
| first fit | 选择第一个足够大的块 | 不一定               | 快                 | 列表前部容易碎片化 |
| next fit  | 从上次位置继续找     | 不一定               | 搜索更分散         | 效果不稳定         |

------

# 8. 其他改进方式

除了基本策略，还有很多更复杂的分配方法。

------

## 8.1 分离空闲列表 segregated list

分离空闲列表的思想是：

> 为常见大小的对象单独维护空闲列表。

例如程序经常申请：

```text
32B、64B、128B
```

那么分配器可以维护：

```text
32B list  -> [32B][32B][32B]
64B list  -> [64B][64B]
128B list -> [128B][128B]
普通 list -> 处理其他大小请求
```

### 优点

分配和释放速度快。

因为不用遍历复杂列表，直接去对应大小的列表中取。

### 缺点

需要决定每种大小的列表应该预留多少空间。

预留太多会浪费，预留太少又不够用。

------

## 8.2 slab allocator 厚块分配器

`slab allocator` 是分离空闲列表思想的经典工程实现。

它常用于操作系统内核。

内核中有很多对象会被频繁创建和销毁，例如：

```text
锁
inode
文件对象
进程结构
网络包结构
```

slab allocator 会为这些常用对象建立对象缓存：

```text
inode cache
lock cache
file cache
```

每个缓存中保存固定类型、固定大小的对象。

------

### slab allocator 为什么快？

因为它让空闲对象保持在预初始化状态。

普通方式：

```text
申请内存
初始化对象
使用对象
销毁对象
释放内存
```

slab 方式：

```text
从缓存拿一个已初始化对象
使用对象
放回缓存
```

这样可以避免频繁初始化和销毁对象的开销。

------

## 8.3 伙伴系统 buddy system

伙伴系统的目标是：

> 让合并操作变得简单。

它将内存按照 2 的幂次大小管理：

```text
64KB
32KB
16KB
8KB
4KB
...
```

------

### 伙伴系统如何分配？

假设初始有 64KB 空间：

```text
[64KB]
```

现在申请 7KB。

系统会分配能容纳 7KB 的最小 2 的幂大小，也就是 8KB。

于是不断二分：

```text
[64KB]
   ↓
[32KB][32KB]
   ↓
[16KB][16KB][32KB]
   ↓
[8KB][8KB][16KB][32KB]
```

然后分配其中一个 8KB 块。

------

### 伙伴系统如何合并？

每次分裂出来的两个块互为伙伴：

```text
[16KB]
   ↓
[8KB buddy A][8KB buddy B]
```

释放时，如果一个块的伙伴也是空闲的，就可以合并：

```text
8KB + 8KB -> 16KB
```

然后继续向上检查：

```text
16KB + 16KB -> 32KB
32KB + 32KB -> 64KB
```

直到无法继续合并。

------

### 伙伴系统的优点

合并简单，伙伴关系可以通过地址计算出来。

### 伙伴系统的缺点

容易产生内部碎片。

例如：

```text
申请 9KB
实际分配 16KB
浪费 7KB
```

因为伙伴系统只能分配 2 的幂次大小的块。

------

## 8.4 更复杂的数据结构

简单空闲列表的查找可能很慢。

因此，更高级的分配器可能使用：

```text
平衡二叉树
伸展树
偏序树
```

这些数据结构可以加快查找速度。

本质是：

> 用更复杂的实现，换取更好的性能。

------

## 8.5 多核与多线程优化

现代程序通常是多线程的。

如果所有线程都访问同一个全局空闲列表，就会出现严重锁竞争。

例如：

```text
线程1 malloc
线程2 malloc
线程3 malloc
线程4 malloc
```

如果它们都需要抢同一把锁：

```text
lock
操作全局空闲列表
unlock
```

性能会下降。

所以现代分配器通常会使用：

```text
每线程缓存
每 CPU 缓存
局部空闲列表
减少全局锁竞争
```

这样可以提升多核系统上的分配性能。

------

# 9. 本章整体总结

本章讨论了可变大小内存分配中的核心问题。

主线如下：

```text
问题：可变大小分配会产生外部碎片
    ↓
机制：分割、合并、header、空闲列表
    ↓
策略：best fit、worst fit、first fit、next fit
    ↓
优化：分离空闲列表、slab、伙伴系统
    ↓
现实挑战：速度、碎片、空间利用率、多线程扩展性
```

# 10. 重点概念总结

## 10.1 分割

当空闲块大于请求大小时，将它切成两部分：

```text
一部分分配给用户
另一部分继续保留为空闲块
```

------

## 10.2 合并

释放内存时，如果相邻块也是空闲的，就合并成更大的空闲块。

目的是减少外部碎片。

------

## 10.3 header

分配器在用户块前面保存元数据，用来记录块大小等信息。

这样 `free(ptr)` 才能知道释放多大的空间。

------

## 10.4 空闲列表

用于记录当前所有空闲块。

空闲列表节点通常直接嵌入在空闲块内部。

------

## 10.5 best fit

选择能满足请求的最小空闲块。

空间浪费少，但查找慢，容易产生小碎片。

------

## 10.6 first fit

选择第一个能满足请求的空闲块。

速度快，但列表前部容易碎片化。

------

## 10.7 segregated list

为常见大小的对象维护独立空闲列表。

适合频繁分配固定大小对象的场景。

------

## 10.8 slab allocator

为常用内核对象建立对象缓存。

对象可以保持预初始化状态，从而减少初始化和销毁开销。

------

## 10.9 buddy system

按照 2 的幂次大小分配内存。

优点是合并简单，缺点是可能产生较多内部碎片。

------

# 11. 常见问题

## 问：为什么 free 不需要传入大小？

因为分配器在用户块前面保存了 header，里面记录了块大小。

`free(ptr)` 时，分配器可以通过指针运算找到 header，从而知道该释放多大的空间。

------

## 问：什么是外部碎片？

空闲空间总量足够，但被分散成多个小块，导致无法满足大的连续空间请求。

------

## 问：什么是内部碎片？

分配器实际给出的空间大于用户请求的空间，多出来但无法被用户有效使用的部分就是内部碎片。

------

## 问：为什么需要合并？

如果释放后的相邻空闲块不合并，堆会变得越来越碎，导致大块内存申请失败。

------

## 问：伙伴系统为什么容易合并？

因为每个块的伙伴可以通过地址关系快速确定。

如果伙伴空闲，就可以直接合并。

# 13. 一句话总结

空闲空间管理的本质是：

> 在可变大小内存分配中，通过 header 记录块信息，通过空闲列表追踪空闲块，通过分割满足请求，通过合并减少碎片，再用不同策略和数据结构在速度、空间利用率和实现复杂度之间做权衡。
