# 第28章 锁（Locks）总结

## 1. 为什么需要锁？

多线程程序中，多个线程可能同时访问共享变量，例如：

```c
balance = balance + 1;
```

这条语句看似简单，但实际可能分成：

```text
读取 balance
加 1
写回 balance
```

如果两个线程交替执行，就可能发生竞态条件，导致结果错误。

因此需要一种机制保证：

```text
同一时刻只能有一个线程进入临界区
```

这种机制就是 **锁（Lock）**。

------

## 2. Pthread 锁：互斥量 mutex

POSIX 线程库中，锁被称为 **互斥量（mutex）**。

基本用法：

```c
pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;

pthread_mutex_lock(&lock);

balance = balance + 1;

pthread_mutex_unlock(&lock);
```

含义：

```text
加锁
进入临界区
修改共享变量
解锁
```

只要所有线程都遵守这个规则，就可以保证同一时刻只有一个线程修改共享数据。

------

## 3. 粗粒度锁与细粒度锁

### 3.1 粗粒度锁

粗粒度锁是指使用一把大锁保护大量数据结构。

优点：

```text
实现简单
不容易写错
```

缺点：

```text
并发度低
很多本来可以同时执行的操作也被迫排队
```

例如整个系统只用一把锁，那么所有线程访问任何共享资源都必须等待。

------

### 3.2 细粒度锁

细粒度锁是指不同数据结构使用不同锁。

例如：

```c
pthread_mutex_t balance_lock;
pthread_mutex_t queue_lock;
pthread_mutex_t log_lock;
```

优点：

```text
并发度高
不同线程访问不同资源时可以同时进行
```

缺点：

```text
实现复杂
容易出现死锁
调试难度更大
```

------

## 4. 如何评价一个锁？

一个好的锁主要看三个方面。

------

### 4.1 正确性

锁必须保证：

```text
一次只有一个线程进入临界区
```

这是锁最基本的要求。

------

### 4.2 公平性

锁应该尽量避免某个线程一直抢不到锁。

如果某个线程长期无法获得锁，就称为：

```text
饥饿 starvation
```

------

### 4.3 性能

锁本身不能带来过高开销。

需要考虑三种情况：

```text
1. 没有竞争时，加锁和解锁是否很快
2. 单 CPU 上多个线程竞争锁时，开销如何
3. 多 CPU 上多个线程竞争锁时，开销如何
```

------

## 5. 关闭中断实现锁

最早的一种方法是在进入临界区时关闭中断：

```c
void lock() {
    DisableInterrupts();
}

void unlock() {
    EnableInterrupts();
}
```

在单处理器系统上，关闭中断后，线程不会被时钟中断打断，因此可以保证临界区代码连续执行。

------

### 5.1 关闭中断的问题

这种方法现在不能作为通用锁，原因有：

```text
1. 普通应用程序不能拥有关闭中断的权限
2. 多处理器系统上无效
3. 可能导致中断丢失
4. 关闭和打开中断本身开销较高
```

尤其是在多 CPU 上，一个 CPU 关闭中断，并不能阻止其他 CPU 上的线程进入临界区。

因此关闭中断只适合操作系统内核在极少数场景下短时间使用。

------

## 6. 简单 flag 锁为什么失败？

一种直觉写法是：

```c
while (flag == 1)
    ;

flag = 1;
```

看起来好像可以实现锁，但实际上不正确。

原因是：

```text
检查 flag
设置 flag
```

这两个操作不是原子的。

可能发生如下情况：

```text
初始 flag = 0

线程 A 检查 flag，发现是 0
线程 A 被切走

线程 B 检查 flag，发现也是 0
线程 B 设置 flag = 1，进入临界区

线程 A 恢复执行
线程 A 也设置 flag = 1，进入临界区
```

结果两个线程同时进入临界区，互斥失败。

------

## 7. Test-and-Set 自旋锁

为了正确实现锁，需要硬件提供原子指令。

Test-and-Set 的伪代码：

```c
int TestAndSet(int *old_ptr, int new) {
    int old = *old_ptr;
    *old_ptr = new;
    return old;
}
```

它会原子地完成：

```text
读取旧值
写入新值
返回旧值
```

------

### 7.1 用 Test-and-Set 实现锁

```c
typedef struct lock_t {
    int flag;
} lock_t;

void init(lock_t *lock) {
    lock->flag = 0;
}

void lock(lock_t *lock) {
    while (TestAndSet(&lock->flag, 1) == 1)
        ;
}

void unlock(lock_t *lock) {
    lock->flag = 0;
}
```

如果 `flag == 0`，说明锁空闲，线程可以获得锁。

如果 `flag == 1`，说明锁已经被持有，线程会一直循环等待。

这种锁叫：

```text
自旋锁 Spin Lock
```

因为拿不到锁的线程不会睡眠，而是一直原地循环。

------

### 7.2 自旋锁的优缺点

优点：

```text
实现简单
可以正确实现互斥
多 CPU 且临界区很短时效果不错
```

缺点：

```text
不保证公平
可能导致线程饥饿
单 CPU 上性能很差
等待时会浪费 CPU
```

------

## 8. Compare-and-Swap：比较并交换

CAS 的伪代码：

```c
int CompareAndSwap(int *ptr, int expected, int new) {
    int actual = *ptr;
    if (actual == expected)
        *ptr = new;
    return actual;
}
```

含义是：

```text
如果 *ptr 等于 expected，就把它改成 new
否则什么也不做
最后返回原来的实际值
```

------

### 8.1 用 CAS 实现锁

```c
void lock(lock_t *lock) {
    while (CompareAndSwap(&lock->flag, 0, 1) == 1)
        ;
}
```

CAS 和 Test-and-Set 都可以实现自旋锁。

不过 CAS 更强大，因为它可以表达：

```text
只有当变量仍然是我预期的旧值时，才更新它
```

所以 CAS 不仅能实现锁，还能实现无锁数据结构。

------

## 9. LL/SC：链接加载与条件存储

有些处理器提供另一组原子操作：

```text
Load-Linked
Store-Conditional
```

简称：

```text
LL / SC
```

------

### 9.1 LoadLinked

```c
int LoadLinked(int *ptr) {
    return *ptr;
}
```

它读取内存中的值，并记录这次读取的地址。

------

### 9.2 StoreConditional

```c
int StoreConditional(int *ptr, int value) {
    if (no one has updated *ptr since the LoadLinked) {
        *ptr = value;
        return 1;
    } else {
        return 0;
    }
}
```

如果从上次 LoadLinked 之后没有其他线程修改过该地址，那么 StoreConditional 成功。

否则失败。

------

### 9.3 用 LL/SC 实现锁

```c
void lock(lock_t *lock) {
    while (1) {
        while (LoadLinked(&lock->flag) == 1)
            ;

        if (StoreConditional(&lock->flag, 1) == 1)
            return;
    }
}

void unlock(lock_t *lock) {
    lock->flag = 0;
}
```

LL/SC 可以保证，即使多个线程同时看到 `flag == 0`，最终也只有一个线程的 StoreConditional 能成功。

------

## 10. Fetch-and-Add 与 Ticket 锁

Fetch-and-Add 的伪代码：

```c
int FetchAndAdd(int *ptr) {
    int old = *ptr;
    *ptr = old + 1;
    return old;
}
```

它会原子地：

```text
返回旧值
让变量加 1
```

------

### 10.1 Ticket 锁

Ticket 锁类似银行排队取号。

```c
typedef struct lock_t {
    int ticket;
    int turn;
} lock_t;
```

其中：

```text
ticket：发号机
turn：当前叫到的号码
```

------

### 10.2 加锁

```c
void lock(lock_t *lock) {
    int myturn = FetchAndAdd(&lock->ticket);

    while (lock->turn != myturn)
        ;
}
```

线程先拿到自己的号码 `myturn`。

只有当：

```text
lock->turn == myturn
```

时，线程才能进入临界区。

------

### 10.3 解锁

```c
void unlock(lock_t *lock) {
    FetchAndAdd(&lock->turn);
}
```

释放锁时，`turn` 加 1，相当于叫下一个号。

------

### 10.4 Ticket 锁的优点

Ticket 锁最大的优点是：

```text
公平
```

线程按照取号顺序进入临界区，避免了普通自旋锁可能导致的饥饿问题。

缺点是：

```text
仍然需要自旋等待
```

------

## 11. 自旋过多的问题

自旋锁虽然正确，但可能浪费 CPU。

尤其在单 CPU 上：

```text
线程 A 持有锁
线程 A 被切走
线程 B 运行，想获取锁
线程 B 发现锁被占用，于是一直自旋
```

问题是，释放锁的线程 A 现在根本没有运行。

所以线程 B 自旋没有任何意义，只是在浪费 CPU 时间。

如果有 N 个线程竞争同一把锁，可能浪费 N-1 个时间片。

------

## 12. yield：主动让出 CPU

一种简单改进是：拿不到锁就调用 `yield()`。

```c
void lock() {
    while (TestAndSet(&flag, 1) == 1)
        yield();
}
```

`yield()` 的作用是：

```text
当前线程主动放弃 CPU
从 running 状态变成 ready 状态
等待调度器再次调度
```

------

### 12.1 yield 的优点

```text
减少纯自旋带来的 CPU 浪费
```

------

### 12.2 yield 的缺点

如果有很多线程竞争锁，它们可能不断：

```text
运行
发现锁被占用
yield
再次被调度
再次 yield
```

这样会产生大量上下文切换。

而且 yield 不能保证公平性，仍然可能出现饥饿。

------

## 13. 使用队列：休眠替代自旋

更好的办法是：

```text
拿不到锁的线程加入等待队列
然后睡眠
锁释放时唤醒等待队列中的线程
```

这需要操作系统支持。

Solaris 提供了：

```c
park();
unpark(threadID);
```

其中：

```text
park：让当前线程睡眠
unpark(threadID)：唤醒指定线程
```

------

## 14. 基于队列的锁

结构如下：

```c
typedef struct lock_t {
    int flag;
    int guard;
    queue_t *q;
} lock_t;
```

含义：

```text
flag：表示锁是否被持有
guard：保护锁内部数据结构的小锁
q：等待锁的线程队列
```

------

### 14.1 lock 过程

```c
void lock(lock_t *m) {
    while (TestAndSet(&m->guard, 1) == 1)
        ;

    if (m->flag == 0) {
        m->flag = 1;
        m->guard = 0;
    } else {
        queue_add(m->q, gettid());
        m->guard = 0;
        park();
    }
}
```

逻辑：

```text
先获取 guard
如果锁空闲，直接获得锁
如果锁被占用，把自己加入等待队列，然后睡眠
```

------

### 14.2 unlock 过程

```c
void unlock(lock_t *m) {
    while (TestAndSet(&m->guard, 1) == 1)
        ;

    if (queue_empty(m->q))
        m->flag = 0;
    else
        unpark(queue_remove(m->q));

    m->guard = 0;
}
```

逻辑：

```text
如果等待队列为空，直接释放锁
如果等待队列不为空，唤醒队列中的一个线程
```

------

## 15. guard 的作用

`guard` 是锁内部的小自旋锁，用来保护：

```text
flag
queue
```

因为多个线程可能同时修改锁内部状态。

不过 `guard` 保护的代码很短，只是 lock/unlock 里的几条指令，所以短暂自旋可以接受。

------

## 16. 为什么 unlock 时不把 flag 设为 0？

当等待队列不为空时，unlock 中会：

```c
unpark(queue_remove(m->q));
```

但不会执行：

```c
m->flag = 0;
```

原因是：

```text
锁直接从当前线程传递给被唤醒的线程
```

如果先把 `flag` 设置为 0，其他新来的线程可能会抢走锁，导致等待队列中的线程被插队。

所以这里保持 `flag == 1`，表示锁仍然被持有，只是所有权转移给下一个线程。

------

## 17. wakeup/waiting race：唤醒等待竞争

基于 park/unpark 的锁存在一个隐蔽问题。

假设线程 B 准备睡眠：

```c
queue_add(m->q, gettid());
m->guard = 0;
park();
```

如果 B 在执行 `park()` 之前被切走。

然后线程 A 释放锁，执行：

```c
unpark(B);
```

这时 B 还没真正睡眠，唤醒可能丢失。

之后 B 恢复执行，调用：

```c
park();
```

结果 B 真的睡着了，但已经没人会再唤醒它。

这种问题叫：

```text
wakeup/waiting race
```

也就是：

```text
唤醒发生在睡眠之前，导致唤醒丢失
```

------

## 18. setpark 解决唤醒丢失

Solaris 增加了：

```c
setpark();
```

修改后的逻辑是：

```c
queue_add(m->q, gettid());
setpark();
m->guard = 0;
park();
```

`setpark()` 表示：

```text
当前线程马上要 park
```

如果在真正 `park()` 之前已经有人调用了 `unpark()`，后面的 `park()` 会直接返回，不会永久睡眠。

------

## 19. Linux futex

Linux 提供了 futex。

futex 可以理解为：

```text
Fast User-space Mutex
```

核心思想是：

```text
无竞争时，在用户态完成加锁和解锁
有竞争时，进入内核睡眠或唤醒
```

------

### 19.1 futex_wait

```c
futex_wait(address, expected)
```

含义：

```text
如果 address 处的值等于 expected，就让当前线程睡眠
否则立刻返回
```

这个检查和睡眠由内核保证原子性，因此可以避免唤醒丢失问题。

------

### 19.2 futex_wake

```c
futex_wake(address)
```

含义：

```text
唤醒等待 address 这个内存地址的一个线程
```

Linux 会为某个 futex 地址维护等待队列。

------

### 19.3 futex 的优点

futex 的优点是：

```text
无竞争时非常快
有竞争时不会浪费 CPU 自旋
只在必要时进入内核
```

这也是现代锁实现的重要思想。

------

## 20. 两阶段锁

两阶段锁结合了自旋和睡眠。

基本思想：

```text
第一阶段：先自旋一段时间
第二阶段：如果还拿不到锁，就睡眠
```

------

### 20.1 为什么先自旋？

因为很多临界区非常短。

如果线程刚拿不到锁就睡眠，会产生：

```text
系统调用开销
上下文切换开销
之后再被唤醒的开销
```

可能比短暂自旋还贵。

所以先自旋一小会儿，赌锁很快会释放。

------

### 20.2 为什么之后要睡眠？

如果自旋一段时间仍然拿不到锁，说明锁可能不会很快释放。

继续自旋会浪费 CPU。

因此进入睡眠更合理。

------

### 20.3 两阶段锁的本质

两阶段锁是一个混合方案：

```text
短等待：自旋
长等待：睡眠
```

真实系统中的锁通常都会使用类似思想。

------

## 21. 第28章整体发展路线

本章锁的实现路线可以总结为：

```text
关闭中断
    ↓
简单 flag 锁
    ↓
Test-and-Set 自旋锁
    ↓
CAS / LL-SC 等原子指令
    ↓
Fetch-and-Add 实现 Ticket 锁
    ↓
yield 减少自旋浪费
    ↓
park/unpark + 等待队列
    ↓
Linux futex
    ↓
两阶段锁
```

------

## 22. 各种锁方法对比

| 方法               | 是否正确互斥 | 公平性     | 性能特点             | 主要问题                 |
| ------------------ | ------------ | ---------- | -------------------- | ------------------------ |
| 关闭中断           | 单 CPU 可行  | 一般       | 简单                 | 多 CPU 无效，权限要求高  |
| 简单 flag          | 不正确       | 无         | 看似简单             | test 和 set 不是原子操作 |
| Test-and-Set       | 正确         | 差         | 多核短临界区较好     | 会自旋浪费 CPU           |
| CAS                | 正确         | 差         | 和 TAS 类似          | 简单锁场景下仍自旋       |
| LL/SC              | 正确         | 差         | 依赖硬件支持         | 仍可能自旋               |
| Ticket 锁          | 正确         | 好         | 按顺序排队           | 仍然自旋                 |
| TAS + yield        | 正确         | 差         | 减少空转             | 上下文切换开销大         |
| park/unpark 队列锁 | 正确         | 较好       | 避免长时间自旋       | 实现复杂，需 OS 支持     |
| futex              | 正确         | 较好       | 无竞争快，有竞争睡眠 | 实现复杂                 |
| 两阶段锁           | 正确         | 视实现而定 | 折中性能好           | 参数调优困难             |

## 23. 考试重点

### 23.1 为什么简单 flag 锁错误？

因为：

```text
检查 flag 和设置 flag 不是原子操作
```

中间可能发生线程切换，导致多个线程同时进入临界区。

------

### 23.2 Test-and-Set 的作用是什么？

它原子地完成：

```text
读取旧值
写入新值
返回旧值
```

可以用来实现自旋锁。

------

### 23.3 CAS 的作用是什么？

CAS 原子地完成：

```text
如果内存值等于 expected，就更新为 new
否则不更新
```

它比 Test-and-Set 更通用。

------

### 23.4 Ticket 锁为什么公平？

因为它像银行排队：

```text
每个线程先取号
按照 turn 的顺序进入临界区
```

所以先来的线程先获得锁。

------

### 23.5 自旋锁什么时候性能好？

自旋锁适合：

```text
多 CPU
临界区很短
锁很快释放
```

不适合：

```text
单 CPU
临界区很长
持锁线程可能被切走
```

------

### 23.6 yield 能解决什么问题？

yield 可以减少纯自旋带来的 CPU 浪费。

但它不能彻底解决：

```text
上下文切换开销
公平性问题
饥饿问题
```

------

### 23.7 为什么需要等待队列？

等待队列可以：

```text
让拿不到锁的线程睡眠
避免浪费 CPU
控制锁释放后唤醒谁
提高公平性
```

------

### 23.8 guard 的作用是什么？

`guard` 是锁内部的小锁，用来保护：

```text
flag
queue
```

它不是保护用户临界区，而是保护锁实现自身的数据结构。

------

### 23.9 wakeup/waiting race 是什么？

它指：

```text
线程准备睡眠
但还没真正睡眠时，被别人提前唤醒
之后线程真正睡眠
结果唤醒丢失，线程可能永远睡下去
```

------

### 23.10 futex 的核心思想是什么？

```text
无竞争时用户态快速完成
有竞争时进入内核睡眠或唤醒
```

------

### 23.11 两阶段锁的核心思想是什么？

```text
先自旋一段时间
如果还拿不到锁，再睡眠
```

它结合了自旋锁和阻塞锁的优点。

## 24. 一句话总结

锁的实现依赖两类支持：

```text
硬件原子指令：保证锁的正确性
操作系统睡眠/唤醒机制：减少自旋浪费，提高性能
```

真实系统中的高性能锁，通常不是单纯自旋，也不是单纯睡眠，而是：

```text
先快速尝试
失败后短暂自旋
再失败就进入内核睡眠
锁释放时唤醒等待线程
```

这就是现代锁实现的核心思想。
