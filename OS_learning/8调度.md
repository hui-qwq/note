# 第 8 章 调度：多级反馈队列 MLFQ 总结

## 1. 本章重要性

多级反馈队列（Multi-Level Feedback Queue，MLFQ）是操作系统调度中非常重要的一种思想。

它的重要性在于：

- 它比 FIFO、SJF、RR 更接近真实操作系统中的调度策略；
- 它能在不知道作业运行时间的情况下，尽量兼顾响应时间和周转时间；
- 它体现了操作系统中的一个重要思想：根据历史行为预测未来行为；
- 它常考简答题、规则题和调度过程模拟题。

一句话概括：

> MLFQ 是一种动态优先级调度算法，通过观察进程过去的行为来调整其优先级，从而兼顾短任务、交互任务和长时间 CPU 密集型任务。

------

## 2. MLFQ 要解决的问题

前面学过的调度算法各有问题：

| 调度算法 | 优点       | 缺点                 |
| -------- | ---------- | -------------------- |
| FIFO     | 简单       | 短任务可能等待很久   |
| SJF/STCF | 周转时间好 | 需要提前知道作业长度 |
| RR       | 响应时间好 | 周转时间可能较差     |

现实中，操作系统通常不知道一个进程会运行多久。

所以 MLFQ 的目标是：

1. 不知道作业长度时，尽量接近 SJF/STCF 的效果；
2. 给交互型任务较好的响应时间；
3. 让长时间运行的 CPU 密集型任务也能持续推进；
4. 防止某些进程长期得不到 CPU；
5. 防止进程欺骗调度器。

------

## 3. MLFQ 的基本思想

MLFQ 有多个优先级队列。

例如：

```text
Q2：最高优先级
Q1：中等优先级
Q0：最低优先级
```

调度规则是：

- 高优先级队列中的进程先运行；
- 同一优先级队列中的进程采用轮转调度；
- 新进程先进入最高优先级队列；
- 如果一个进程长期占用 CPU，就降低它的优先级；
- 如果一个进程经常主动放弃 CPU，说明它可能是交互型任务，应给予较高优先级；
- 系统会周期性提升所有进程优先级，防止饥饿。

------

## 4. MLFQ 的五条核心规则

### 规则 1：高优先级先运行

如果 A 的优先级大于 B：

```text
运行 A，不运行 B
```

也就是说，只要高优先级队列中有进程，低优先级队列中的进程就不能运行。

------

### 规则 2：同优先级使用轮转调度

如果 A 和 B 的优先级相同：

```text
A 和 B 轮转运行
```

例如：

```text
Q2: A B
```

执行顺序可能是：

```text
A → B → A → B → ...
```

------

### 规则 3：新工作进入最高优先级队列

新进程刚进入系统时，放到最高优先级队列。

原因是：

> 操作系统不知道它是短任务还是长任务，所以先假设它是短任务。

如果它真的是短任务，就可以很快完成；
如果它是长任务，之后会因为占用 CPU 太久而被降级。

------

### 规则 4：用完某一层时间配额就降级

如果一个进程在某一层队列中用完了自己的 CPU 时间配额，就降低它的优先级。

重点是：

```text
看累计 CPU 使用时间，而不是只看单次时间片有没有用完。
```

这样可以防止进程通过频繁主动 I/O 来欺骗调度器。

------

### 规则 5：周期性提升所有进程优先级

经过一段时间 S 后，把系统中所有进程重新放回最高优先级队列。

作用：

1. 防止低优先级进程饿死；
2. 让已经变成交互型的进程重新获得较好的响应时间；
3. 使调度器可以重新观察进程行为。

------

## 5. MLFQ 如何近似 SJF？

SJF 的思想是：

```text
短作业优先
```

但是 SJF 需要提前知道作业长度。

MLFQ 不知道作业长度，所以它采用一种猜测策略：

```text
新进程先放最高优先级；
如果它很快完成，就说明它确实像短任务；
如果它一直占用 CPU，就逐渐降低优先级。
```

因此，MLFQ 可以在不知道作业长度的情况下，近似实现短作业优先。

------

## 6. MLFQ 如何照顾交互型任务？

交互型任务通常具有这样的特点：

```text
运行一小会儿 → 等待 I/O → 再运行一小会儿 → 再等待 I/O
```

例如：

- 键盘输入；
- 鼠标点击；
- 文本编辑器；
- 终端命令；
- 图形界面程序。

这些任务不一定需要很多 CPU 时间，但非常需要快速响应。

MLFQ 会根据进程行为判断：

```text
如果进程经常主动放弃 CPU，它可能是交互型任务。
```

因此，MLFQ 会让这类进程保持较高优先级，从而改善响应时间。

------

## 7. MLFQ 的问题与解决方法

### 问题 1：饥饿

如果系统中有大量交互型任务，它们一直留在高优先级队列中，那么低优先级的长任务可能永远得不到 CPU。

解决方法：

```text
周期性提升所有进程优先级。
```

也就是规则 5。

------

### 问题 2：调度器可能被欺骗

旧规则中，如果一个进程在时间片用完前主动释放 CPU，它就可以保持高优先级。

于是恶意进程可以这样做：

```text
每次快用完时间片时，主动发起一次 I/O。
```

这样它就能一直留在高优先级队列，获得远超公平的 CPU 时间。

解决方法：

```text
累计计算进程在某一层队列中使用的 CPU 时间。
```

只要累计时间用完了该层配额，就降级。

------

### 问题 3：进程行为会变化

一个进程可能一开始是 CPU 密集型，后来变成交互型。

如果它之前已经被降到低优先级，后面就很难得到快速响应。

解决方法：

```text
周期性提升所有进程优先级。
```

这样进程有机会重新回到高优先级，让调度器重新观察它的行为。

------

### 问题 4：参数难以调优

MLFQ 有很多参数：

- 设置多少个队列？
- 每个队列时间片多长？
- 多久进行一次优先级提升？
- I/O 完成后进程放队头还是队尾？

这些参数没有绝对标准答案，需要根据系统负载调优。

例如：

```text
高优先级队列：时间片较短，适合交互任务
低优先级队列：时间片较长，适合 CPU 密集型任务
```

------

## 8. 时间片如何配置？

一般来说：

```text
优先级越高，时间片越短；
优先级越低，时间片越长。
```

原因：

- 高优先级队列主要服务交互型任务，需要快速响应；
- 低优先级队列主要服务 CPU 密集型任务，较长时间片可以减少上下文切换开销。

例如：

```text
Q2：10ms
Q1：20ms
Q0：40ms
```

------

## 9. nice 的含义

有些系统允许用户给调度器一些建议，例如 Unix/Linux 中的 `nice` 命令。

`nice` 值可以影响进程优先级。

一般来说：

```text
nice 值越高，进程越“谦让”，优先级倾向越低；
nice 值越低，进程越“不谦让”，优先级倾向越高。
```

但 `nice` 只是建议，操作系统不一定完全按照它执行。

------

## 10. MLFQ 的优点

MLFQ 的优点包括：

1. 不需要提前知道作业长度；
2. 可以改善短任务的周转时间；
3. 可以改善交互型任务的响应时间；
4. 可以让长任务持续推进；
5. 可以通过优先级提升避免饥饿；
6. 可以通过累计计时防止调度器被欺骗；
7. 思想接近真实操作系统中的调度策略。

------

## 11. MLFQ 的缺点

MLFQ 也有一些缺点：

1. 参数较多，调优困难；
2. 队列数量、时间片长度、提升周期都不好确定；
3. 如果参数设置不好，可能影响公平性或响应时间；
4. 实现比 FIFO、SJF、RR 更复杂；
5. 仍然依赖历史行为预测未来，预测可能出错。

------

## 12. 常见考试问法

### 问题 1：什么是 MLFQ？

答：

> MLFQ 是多级反馈队列调度算法。它设置多个不同优先级的队列，高优先级队列优先运行，同一队列内部采用轮转调度。进程的优先级会根据其运行行为动态调整，从而在不知道作业长度的情况下兼顾响应时间和周转时间。

------

### 问题 2：MLFQ 的基本规则是什么？

答：

```text
1. 如果 A 的优先级高于 B，则运行 A。
2. 如果 A 和 B 优先级相同，则轮转运行。
3. 新工作进入系统时，放入最高优先级队列。
4. 工作用完某一级队列的时间配额后，降低优先级。
5. 经过一段时间 S，将所有工作重新加入最高优先级队列。
```

------

### 问题 3：为什么新进程要放入最高优先级队列？

答：

> 因为操作系统不知道新进程是短任务还是长任务，所以先假设它是短任务。如果它确实是短任务，就能很快完成；如果它是长任务，就会因为持续占用 CPU 而逐渐降级。

------

### 问题 4：MLFQ 如何防止饥饿？

答：

> MLFQ 通过周期性提升所有进程优先级来防止饥饿。经过一段时间 S 后，系统将所有进程重新放入最高优先级队列，使低优先级进程也能定期获得运行机会。

------

### 问题 5：MLFQ 如何防止调度器被欺骗？

答：

> MLFQ 不再只看进程是否用完整个时间片，而是累计进程在某一级队列中使用的 CPU 时间。只要累计时间用完该层配额，不管中途是否主动放弃 CPU，都要降低优先级。

------

### 问题 6：为什么优先级越低，时间片通常越长？

答：

> 高优先级队列主要面向交互型任务，需要短时间片来提高响应速度；低优先级队列主要面向 CPU 密集型任务，使用较长时间片可以减少上下文切换开销，提高 CPU 利用率。

------

## 13. 作业题核心思路

### 如何让 MLFQ 退化成 RR？

只设置一个队列：

```bash
./mlfq.py -n 1 -q 10 -c
```

因为只有一个队列，就没有优先级升降，所有任务都在同一队列中轮转运行。

------

### 如何让进程愚弄旧版 MLFQ？

旧规则下，只要进程在时间片用完前主动 I/O，就不会降级。

所以可以设计：

```text
时间片 = 100ms
进程每运行 99ms 就主动 I/O
```

这样该进程能长期保持高优先级，几乎获得 99% CPU。

------

### 如何保证长任务至少获得 5% CPU？

如果最高队列时间片是 10ms，boost 周期为 S。

要求：

```text
10 / S >= 5%
```

即：

```text
S <= 200ms
```

所以最多每 200ms 进行一次优先级提升。

------

## 14. 本章最重要的一句话

> MLFQ 通过多个优先级队列和动态优先级调整，在不知道作业长度的情况下，尽量兼顾短任务的周转时间、交互任务的响应时间以及长任务的公平性。

# 总结

MLFQ 是本章的核心调度算法，也是理解真实操作系统调度策略的重要入口。

它的本质不是某条具体规则，而是一种思想：

> 操作系统不知道未来，就通过观察过去来动态调整策略。

这也是操作系统中非常常见的工程思想。
