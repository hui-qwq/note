# 第13章 抽象：地址空间

## 一、本章核心

本章开始进入操作系统的第二大主题：

> **内存虚拟化**

CPU 虚拟化让每个进程感觉自己独占 CPU；
内存虚拟化则让每个进程感觉自己拥有一整块独立、连续、私有的内存空间。

这个抽象就叫：

> **地址空间 Address Space**

程序看到的是虚拟地址，真实硬件访问的是物理地址。操作系统和硬件负责完成二者之间的转换。

------

## 二、早期系统

早期系统非常简单，内存中通常只有：

```text
+----------------+
| 操作系统        |
+----------------+
| 一个用户程序     |
+----------------+
| 剩余内存        |
+----------------+
```

那时候用户需求低，机器上通常只运行一个程序。

程序直接使用物理内存，几乎没有抽象，也没有复杂的保护机制。

所以早期操作系统比较简单。

------

## 三、多道程序与时分共享

后来计算机非常昂贵，人们希望更充分地利用机器，于是出现了 **多道程序系统**。

### 1. 多道程序

多道程序的核心思想是：

> 当一个进程等待 I/O 时，CPU 可以切换去运行另一个进程。

这样可以提高 CPU 利用率。

例如：

```text
进程 A 等待磁盘 I/O
↓
CPU 运行进程 B
↓
进程 A 的 I/O 完成后再切回来
```

------

### 2. 时分共享

后来人们又希望程序能够及时响应，于是出现了 **时分共享系统**。

时分共享的核心思想是：

> 多个进程轮流使用 CPU，每个进程运行一小段时间。

例如：

```text
进程 A 运行一会儿
进程 B 运行一会儿
进程 C 运行一会儿
再回到进程 A
```

这样用户会感觉多个程序好像在同时运行。

------

## 四、为什么不能把整个进程换到磁盘？

一种粗糙的时分共享方法是：

1. 让进程 A 占用全部内存运行一会儿；
2. 停止 A；
3. 把 A 的所有内存内容保存到磁盘；
4. 从磁盘加载进程 B 的全部内存；
5. 运行 B。

这个方法可以实现进程切换，但问题是：

> 太慢了。

保存寄存器状态比较快，但把整个内存保存到磁盘非常慢。

所以更好的方法是：

> 多个进程同时驻留在内存中，操作系统只切换 CPU 的执行权。

------

## 五、多个进程同时在内存中的问题

多个进程同时放在物理内存中后，会出现保护问题。

例如：

```text
+----------------+
| 操作系统        |
+----------------+
| 进程 A          |
+----------------+
| 进程 B          |
+----------------+
| 进程 C          |
+----------------+
```

如果没有保护机制，进程 A 可能会：

- 读取进程 B 的数据；
- 修改进程 B 的内存；
- 破坏操作系统本身。

所以操作系统必须提供一种机制，让每个进程都只能访问自己的内存。

这就是地址空间和内存虚拟化要解决的问题。

------

## 六、地址空间是什么？

地址空间是操作系统提供给进程的内存抽象。

简单来说：

> 地址空间就是进程“以为自己拥有的内存”。

每个进程都觉得自己拥有一块从 0 开始的连续内存。

但实际上，进程可能被放在物理内存的任意位置。

------

## 七、地址空间的组成

一个进程的地址空间通常包括三个主要部分：

```text
+----------------+
| 代码区 code     |
+----------------+
| 堆 heap         |
|                |
| 空闲区域        |
|                |
| 栈 stack        |
+----------------+
```

------

### 1. 代码区 Code

代码区存放程序的指令。

例如：

```c
int main() {
    return 0;
}
```

编译之后的机器指令就放在代码区。

代码区通常是静态的，程序运行过程中一般不会继续增长。

------

### 2. 堆 Heap

堆用于动态内存分配。

例如 C 语言中的：

```c
malloc()
```

C++ 中的：

```cpp
new
```

Java 中的：

```java
new Object()
```

这些动态申请的内存通常都来自堆。

堆会随着程序运行而增长或收缩。

------

### 3. 栈 Stack

栈用于函数调用。

它保存：

- 局部变量；
- 函数参数；
- 返回地址；
- 函数调用上下文。

例如：

```c
void f() {
    int x = 10;
}
```

变量 `x` 通常就在栈上。

函数调用越深，栈使用得越多。

------

## 八、堆和栈为什么放在两端？

堆和栈都是运行时可能增长的区域。

所以操作系统通常把它们放在地址空间的两端，让它们朝相反方向增长。

```text
低地址
+----------------+
| 代码区 code     |
+----------------+
| 堆 heap         |
| ↓ 增长          |
|                |
| 空闲区域        |
|                |
| ↑ 增长          |
| 栈 stack        |
+----------------+
高地址
```

这样可以让堆和栈都拥有较大的增长空间。

------

## 九、虚拟地址和物理地址

程序看到的地址叫：

> **虚拟地址 Virtual Address**

硬件真正访问内存时使用的地址叫：

> **物理地址 Physical Address**

例如，进程 A 访问虚拟地址 0：

```text
虚拟地址 0
```

但真实访问的可能是：

```text
物理地址 320KB
```

也就是说：

> 程序以为自己从地址 0 开始运行，但实际上它可能被加载到物理内存的其他位置。

操作系统和硬件会负责完成地址转换。

------

## 十、内存虚拟化的核心问题

本章最重要的问题是：

> 操作系统如何在一块物理内存上，为多个进程构建私有的、连续的、可能很大的地址空间？

换句话说：

```text
每个进程都以为自己独占内存
但实际上所有进程共享物理内存
```

这就是内存虚拟化。

------

## 十一、虚拟内存的三个目标

虚拟内存系统主要有三个目标：

```text
透明性
效率
保护
```

------

### 1. 透明性 Transparency

透明性的意思是：

> 程序感觉不到内存被虚拟化了。

程序运行时以为：

- 自己拥有独立内存；
- 地址空间是连续的；
- 自己可以从地址 0 开始使用内存；
- 不需要知道真实物理内存在哪里。

操作系统和硬件在背后完成所有工作。

------

### 2. 效率 Efficiency

虚拟内存不能太慢，也不能太浪费空间。

效率包括两个方面：

#### 时间效率

地址转换不能让程序明显变慢。

如果每访问一次内存都要花很大代价，程序性能就会很差。

因此系统需要硬件支持，比如后面会学到的：

```text
TLB
```

TLB 可以理解为地址翻译的高速缓存。

#### 空间效率

操作系统为了支持虚拟内存，需要维护一些数据结构，比如页表。

这些结构不能占用太多额外内存。

所以虚拟内存系统也要尽量节省空间。

------

### 3. 保护 Protection

保护是虚拟内存的重要目标。

操作系统要保证：

- 进程 A 不能访问进程 B 的内存；
- 用户程序不能破坏操作系统内存；
- 一个出错的进程不能影响其他进程。

这带来了一个重要特性：

> 隔离 Isolation

每个进程都运行在自己的独立环境中。

------

## 十二、你看到的地址都不是真的

用户程序中打印出来的地址，都是虚拟地址。

例如：

```c
#include <stdio.h>
#include <stdlib.h>

int main(int argc, char *argv[]) {
    printf("location of code : %p\n", (void *) main);
    printf("location of heap : %p\n", (void *) malloc(1));
    int x = 3;
    printf("location of stack : %p\n", (void *) &x);
    return x;
}
```

可能输出：

```text
location of code : 0x1095afe50
location of heap : 0x1096008c0
location of stack : 0x7fff691aea64
```

这些地址看起来像真实地址，但它们都是虚拟地址。

真正的物理地址只有操作系统和硬件知道。

所以要记住：

> 用户程序能看到的地址，几乎都是虚拟地址。

------

## 十三、机制与策略

后面学习虚拟内存时，会经常遇到两个词：

```text
机制 mechanism
策略 policy
```

------

### 1. 机制 Mechanism

机制关注：

> 怎么实现？

例如：

- 虚拟地址如何转换成物理地址？
- 页表怎么查？
- TLB 如何加速？
- 缺页异常如何处理？

机制更偏底层实现。

------

### 2. 策略 Policy

策略关注：

> 做选择时选哪个？

例如：

- 内存不够时换出哪个页面？
- 空闲空间如何管理？
- 页面替换算法如何设计？

策略更偏决策。

------

## 十四、本章小结

本章介绍了操作系统中的重要抽象：

> 地址空间

虚拟内存系统负责为程序提供一个：

```text
巨大、稀疏、私有的地址空间
```

### 1. 巨大

程序感觉自己拥有很大的地址空间。

例如 32 位系统可以拥有约 4GB 虚拟地址空间，64 位系统更大。

------

### 2. 稀疏

地址空间虽然看起来很大，但不一定每一块都真正使用了。

通常只有代码区、堆、栈等部分真正占用空间，中间可能有大量空闲区域。

------

### 3. 私有

每个进程都有自己的地址空间。

进程 A 的虚拟地址 `0x1000` 和进程 B 的虚拟地址 `0x1000` 看起来一样，但可能映射到完全不同的物理地址。

```text
进程 A: 虚拟地址 0x1000 → 物理地址 0x30000
进程 B: 虚拟地址 0x1000 → 物理地址 0x80000
```

所以它们互不影响。

------

## 十六、考试重点

### 1. 地址空间是什么？

地址空间是操作系统提供给进程的内存抽象，是进程看到的虚拟内存空间。

------

### 2. 虚拟内存的三个目标是什么？

虚拟内存的三个目标是：

```text
透明性、效率、保护
```

透明性：程序感觉不到内存被虚拟化。
效率：地址转换不能太慢，也不能浪费太多空间。
保护：防止进程互相访问或破坏，实现隔离。

------

### 3. 用户程序打印出的地址是物理地址吗？

不是。

用户程序打印出的地址是虚拟地址。
物理地址由操作系统和硬件负责管理和转换。

------

### 4. 为什么需要内存保护？

因为多个进程同时驻留在内存中，如果没有保护，一个进程可能访问或修改其他进程甚至操作系统的内存。

内存保护可以提高系统可靠性和安全性。

------

### 5. 机制和策略有什么区别？

机制关注“怎么做”，策略关注“选哪个”。

例如：

```text
地址转换是机制
页面替换算法是策略
```

------

## 十七、一句话总结

> 地址空间是操作系统为每个进程制造的内存假象：每个进程都以为自己拥有一块巨大、连续、私有的内存空间，但实际上操作系统和硬件会把虚拟地址转换成物理地址，并保证进程之间互相隔离。

------

## 十八、后续学习主线

接下来学习虚拟内存时，要抓住这条主线：

```text
程序产生虚拟地址
↓
硬件和操作系统进行地址转换
↓
得到物理地址
↓
访问真实物理内存
```

后续内容包括：

- 基址和界限；
- 分段；
- 分页；
- 页表；
- TLB；
- 缺页异常；
- 页面置换；
- 虚拟内存系统完整实现。

本章只是开头，真正的重点在后面的地址转换机制。
