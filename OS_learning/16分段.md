# 第16章：分段机制总结

## 1. 分段是什么？

分段是对 **基址 / 界限机制** 的泛化。

以前的做法是：

```
一个进程 = 一整块连续的虚拟地址空间
```

但这样会浪费很多内存，因为代码、堆、栈之间有大量未使用空间。

分段的做法是：

```
一个进程 = 代码段 + 堆段 + 栈段
```

每个段都有自己的一组寄存器：

| 信息          | 含义                     |
| ------------- | ------------------------ |
| Base          | 段在物理内存中的起始地址 |
| Bound / Limit | 段的大小                 |
| Protection    | 段的访问权限             |
| Direction     | 段的增长方向             |

分段的核心目的：

> 只把真正使用的段放入物理内存，避免虚拟地址空间中的空洞浪费物理内存。

------

## 2. 一个进程的分段样子

假设一个进程的虚拟地址空间是 `0KB ~ 16KB`：

```
虚拟地址空间

0KB
│
│  代码段 Code
│
2KB
│  未使用
│
4KB
│
│  堆段 Heap
│
6KB
│
│  未使用
│
14KB
│
│  栈段 Stack
│
16KB
```

其中：

| 段     | 虚拟地址范围 | 说明               |
| ------ | ------------ | ------------------ |
| 代码段 | 0KB ~ 2KB    | 存放程序指令       |
| 堆段   | 4KB ~ 6KB    | 动态分配内存       |
| 栈段   | 14KB ~ 16KB  | 函数调用、局部变量 |

注意：

```
堆：从低地址向高地址增长
栈：从高地址向低地址增长
```

------

## 3. 段在物理内存中的安排

分段后，操作系统不需要把整个地址空间连续放进物理内存。

它可以把代码段、堆段、栈段分别放到不同位置。

例如：

| 段     | 物理基址 | 大小 |
| ------ | -------- | ---- |
| 代码段 | 32KB     | 2KB  |
| 堆段   | 34KB     | 2KB  |
| 栈段   | 28KB     | 2KB  |

物理内存可能长这样：

```
物理内存

0KB
│ 操作系统
│
16KB
│ 空闲
│
28KB
│ 栈段
│
30KB
│ 空闲
│
32KB
│ 代码段
│
34KB
│ 堆段
│
36KB
│ 空闲
│
64KB
```

可以看到：

> 虚拟地址空间中代码、堆、栈看起来是固定的，但它们在物理内存中可以分散放置。

------

## 4. 地址转换过程

分段地址转换的核心公式：

```
物理地址 = 段的物理基址 + 段内偏移
```

同时还要检查：

```
段内偏移 < 段界限
```

如果越界，就会触发异常，也就是常见的：

```
Segmentation Fault
段错误
```

------

## 5. 地址转换例子

### 例子一：访问代码段地址 100

代码段：

```
虚拟范围：0KB ~ 2KB
物理基址：32KB
大小：2KB
```

虚拟地址 `100` 在代码段内。

段内偏移：

```
100 - 0 = 100
```

物理地址：

```
32KB + 100 = 32868
```

所以：

```
虚拟地址 100 -> 物理地址 32868
```

------

### 例子二：访问堆地址 4200

堆段：

```
虚拟范围：4KB ~ 6KB
物理基址：34KB
大小：2KB
```

虚拟地址 `4200` 在堆段内。

堆从 `4KB = 4096` 开始，所以段内偏移是：

```
4200 - 4096 = 104
```

物理地址：

```
34KB + 104 = 34920
```

所以：

```
虚拟地址 4200 -> 物理地址 34920
```

注意不能直接算：

```
34KB + 4200
```

因为 `4200` 是整个虚拟地址空间中的地址，不是堆段内偏移。

------

## 6. 如何判断访问的是哪个段？

一种常见方法是：用虚拟地址的高几位表示段号。

假设虚拟地址是 14 位，用最高 2 位表示段号：

| 高两位 | 段     |
| ------ | ------ |
| 00     | 代码段 |
| 01     | 堆段   |
| 10     | 未使用 |
| 11     | 栈段   |

地址格式：

```
[ 段号 2位 ][ 段内偏移 12位 ]
```

例如虚拟地址 `4200` 的二进制可以看作：

```
01 0000 0110 1000
```

其中：

```
01 -> 堆段
0000 0110 1000 -> 段内偏移 104
```

于是硬件知道：

```
使用堆段寄存器进行地址转换
```

------

## 7. 栈为什么特殊？

栈和代码段、堆段不一样。

代码段、堆段一般是：

```
低地址 -> 高地址
```

但是栈是：

```
高地址 -> 低地址
```

所以硬件除了记录 `Base` 和 `Bound`，还需要记录：

```
这个段是正向增长，还是反向增长
```

例如：

| 段     | 物理基址 | 大小 | 增长方向 |
| ------ | -------- | ---- | -------- |
| 代码段 | 32KB     | 2KB  | 正向     |
| 堆段   | 34KB     | 2KB  | 正向     |
| 栈段   | 28KB     | 2KB  | 反向     |

------

## 8. 栈地址转换例子

假设访问虚拟地址：

```
15KB
```

它属于栈段。

栈段的虚拟范围可以看作：

```
14KB ~ 16KB
```

或者按高两位划分，栈段对应：

```
12KB ~ 16KB
```

虚拟地址 `15KB` 的段内偏移是：

```
15KB - 12KB = 3KB
```

但栈是反向增长的，所以要把它变成反向偏移：

```
3KB - 4KB = -1KB
```

栈段物理基址是：

```
28KB
```

所以物理地址为：

```
28KB + (-1KB) = 27KB
```

因此：

```
虚拟地址 15KB -> 物理地址 27KB
```

------

## 9. 两个进程如何安排段？

假设有两个进程：进程 A 和进程 B。

它们看到的虚拟地址空间可以完全一样：

```
进程 A 虚拟地址空间

0KB  ~ 2KB   代码段
4KB  ~ 6KB   堆段
14KB ~ 16KB  栈段
进程 B 虚拟地址空间

0KB  ~ 2KB   代码段
4KB  ~ 6KB   堆段
14KB ~ 16KB  栈段
```

但是在物理内存中，它们可以被放到不同位置。

### 进程 A 的段表

| 段     | 虚拟范围    | 物理基址 | 大小 |
| ------ | ----------- | -------- | ---- |
| 代码段 | 0KB ~ 2KB   | 32KB     | 2KB  |
| 堆段   | 4KB ~ 6KB   | 34KB     | 2KB  |
| 栈段   | 14KB ~ 16KB | 28KB     | 2KB  |

### 进程 B 的段表

| 段     | 虚拟范围    | 物理基址 | 大小 |
| ------ | ----------- | -------- | ---- |
| 代码段 | 0KB ~ 2KB   | 40KB     | 2KB  |
| 堆段   | 4KB ~ 6KB   | 56KB     | 2KB  |
| 栈段   | 14KB ~ 16KB | 52KB     | 2KB  |

物理内存可能是：

```
物理内存

0KB
│ 操作系统
│
16KB
│ 空闲
│
28KB
│ 进程 A 栈段
│
32KB
│ 进程 A 代码段
│
34KB
│ 进程 A 堆段
│
40KB
│ 进程 B 代码段
│
52KB
│ 进程 B 栈段
│
56KB
│ 进程 B 堆段
│
64KB
```

重点：

> 进程 A 和进程 B 的虚拟地址可以一样，但映射到的物理地址可以不同。

------

## 10. 同一个虚拟地址在不同进程中的结果

假设两个进程都访问虚拟地址：

```
4200
```

这个地址属于堆段。

段内偏移：

```
4200 - 4096 = 104
```

对于进程 A：

```
堆段物理基址 = 34KB
物理地址 = 34KB + 104 = 34920
```

对于进程 B：

```
堆段物理基址 = 56KB
物理地址 = 56KB + 104 = 57440
```

所以：

```
进程 A 的虚拟地址 4200 -> 物理地址 34920
进程 B 的虚拟地址 4200 -> 物理地址 57440
```

这说明：

> 虚拟地址本身不决定最终访问哪里，当前进程的段寄存器才决定真正的物理地址。

------

## 11. 上下文切换时发生什么？

每个进程都有自己的段寄存器。

CPU 运行进程 A 时：

```
MMU 使用进程 A 的段寄存器
```

CPU 切换到进程 B 时，操作系统必须：

```
保存进程 A 的段寄存器
恢复进程 B 的段寄存器
```

否则进程 B 可能会访问到进程 A 的物理内存，破坏隔离。

------

## 12. 支持共享

分段还可以支持段共享。

最典型的是：

```
共享代码段
```

如果进程 A 和进程 B 运行的是同一个程序，那么它们的代码段内容相同。

操作系统可以让它们的代码段指向同一块物理内存：

```
进程 A 代码段 -> 物理地址 32KB
进程 B 代码段 -> 物理地址 32KB
```

但堆和栈一般不能共享：

```
进程 A 堆段 -> A 自己的物理内存
进程 B 堆段 -> B 自己的物理内存

进程 A 栈段 -> A 自己的物理内存
进程 B 栈段 -> B 自己的物理内存
```

因为堆和栈保存的是每个进程自己的运行数据。

为了安全共享代码段，需要保护位：

| 段     | 权限      |
| ------ | --------- |
| 代码段 | 读 + 执行 |
| 堆段   | 读 + 写   |
| 栈段   | 读 + 写   |

代码段不能写，否则一个进程修改代码，会影响其他共享该代码段的进程。

------

## 13. 分段的优点

分段的优点：

1. 比单一的基址/界限机制更灵活。
2. 只需要把真正使用的段放入物理内存。
3. 能支持稀疏地址空间。
4. 地址转换速度快，适合硬件实现。
5. 可以支持代码段共享。
6. 可以给不同段设置不同权限。

------

## 14. 分段的问题：外部碎片

分段最大的问题是：

```
外部碎片 external fragmentation
```

因为每个段大小不同，物理内存运行一段时间后可能变成：

```
已分配
空闲
已分配
空闲
已分配
空闲
```

虽然总空闲空间很多，但不连续。

例如：

```
空闲 8KB
空闲 4KB
空闲 12KB
```

总共有：

```
8KB + 4KB + 12KB = 24KB
```

但是如果现在需要一个连续的 `20KB` 段，就无法分配。

因为没有一块连续空间达到 `20KB`。

这就是外部碎片的麻烦：

> 总空间够，但连续空间不够。

------

## 15. 如何缓解外部碎片？

一种方法是内存紧凑：

```
把分散的已分配段移动到一起
整理出一大块连续空闲空间
```

整理前：

```
已分配
空闲
已分配
空闲
已分配
空闲
```

整理后：

```
已分配
已分配
已分配
空闲
空闲
空闲
```

但是紧凑的代价很高：

```
需要复制大量内存
需要修改段寄存器
会占用大量 CPU 时间
```

还有一些空闲空间管理算法：

| 算法            | 思想                          |
| --------------- | ----------------------------- |
| First-Fit       | 找第一个足够大的空闲块        |
| Best-Fit        | 找最接近需求大小的空闲块      |
| Worst-Fit       | 找最大的空闲块进行切分        |
| Buddy Algorithm | 伙伴算法，按 2 的幂次管理内存 |

但是这些算法只能缓解外部碎片，不能彻底消除。

------

## 16. 分段的局限

分段虽然比单一基址/界限机制更好，但仍然不够完美。

主要问题：

```
如果一个段内部也很稀疏，分段仍然会浪费内存
```

例如，一个堆段很大：

```
4KB ~ 1GB
```

但实际只用了其中很小一部分。

分段仍然会把整个堆段当成一个整体管理，这就不够灵活。

所以后面需要新的机制：

```
分页 Paging
```

------

## 17. 总结

分段的本质：

```
把一个进程的地址空间拆成多个逻辑段
每个段单独进行地址转换
```

核心公式：

```
物理地址 = 段基址 + 段内偏移
```

核心检查：

```
段内偏移 < 段界限
访问权限合法
```

核心优点：

```
节省内存
支持稀疏地址空间
支持代码共享
支持不同段的权限保护
```

核心缺点：

```
会产生外部碎片
段大小不同，内存管理复杂
无法很好处理段内部稀疏的情况
```

学习路线可以这样记：

```
基址/界限：一个进程一整块
分段：一个进程分成代码、堆、栈几块
分页：一个进程切成很多固定大小的小页
```

一句话压缩：

> 分段让进程的代码、堆、栈可以分别映射到物理内存的不同位置，从而节省空间并支持共享；但因为段大小不固定，会产生外部碎片，所以后来需要分页机制来进一步解决问题。
